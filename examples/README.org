#+title: Nelix — Examples
#+author: zawatton

* What this is

Copy-pasteable =pkg-define= recipes for every supported source +
build-system combination.  Use as a starting point for your own
=Nelix= package definitions.  The examples still use =pkg-define= and
=pkg-install= because those short forms remain the stable DSL surface;
=nelix-define= and =nelix-install= are also available for configs that
prefer the project prefix.

Each file is a self-contained Elisp library.  Load it from your
Emacs config (or =M-x load-file=), then call =(pkg-install
'<package>)= or =(nelix-install '<package>)= to actually build /
install through Nix.

* Hashes — real vs placeholder

Phase 4-H (2026-05-06) shipped *real* prefetched hashes for
four recipes — they are smoke-verified against
=nixpkgs-unstable= on every push (=make smoke-eval=):

| Recipe                | Source hash | Build-system hash         |
|-----------------------+-------------+---------------------------|
| =stdenv-hello.el=     | real        | (none)                    |
| =rust-ripgrep.el=     | real        | =cargoHash= real          |
| =python-black.el=     | real        | (none — pyproject)        |
| =go-hugo.el=          | real        | =vendorHash= real         |

The other recipes (=multi-install=, =emacs-melpa-magit=,
=emacs-trivial-dash=, =private-github=) are still *shape
examples* — they use =sha256-PLACEHOLDER= literals and need
real hashes before =pkg-install=.

When you copy a placeholder recipe, run =pkg-install= once;
Nix will fail with a hash-mismatch error that prints the
*real* hash.  Paste that back and re-install.

For deterministic CI you can also pre-compute hashes:

#+begin_src shell
nix-prefetch-url --type sha256 https://example.com/foo.tar.gz
nix-prefetch-url --unpack https://github.com/owner/repo/archive/REV.tar.gz
nix hash convert --hash-algo sha256 --to sri <base32-hash>
#+end_src

* Caveats

- *Hashes:* the four real-hash recipes are pinned to specific
  =:version= / =:rev= values; bumping either means re-running
  =nix-prefetch-url= and rebuilding the cargo / vendor hash
  via =lib.fakeHash + nix build=.  Placeholder recipes still
  need real hashes before installing to a real profile.
- *Nix daemon:* =pkg-install= shells out to =nix profile install= /
  =nix profile add= via the Phase 1 wrapper.  Make sure flakes are
  enabled and the daemon is running.
- *Source pinning:* the examples pin to a specific =:rev= /
  =:version=.  Bumping the version usually means bumping the hashes
  too.
- *Cargo / vendor hashes:* =rust= and =go= recipes need a *second*
  hash on top of the source hash (=:cargo-sha256= /
  =:vendor-sha256=) — these come from the dependency lock files,
  not the upstream tarball.
- *License field:* =:license= takes a *symbol*, not a string.
  Recognised: =mit=, =gpl2=, =gpl3=, =bsd2=, =bsd3=, =apache2=
  (mapped to =pkgs.lib.licenses.*= internally).  Unknown symbols
  fall back to a quoted string literal, which Nix accepts but does
  not link to its license metadata.
