#+TITLE: Doc 25 - Nelix native/AOT manifest engine
#+DATE: 2026-06-17
#+STATUS: DRAFT

* Problem

Doc 24 made =nelix --runtime nelisp= correct enough for normal
=list=, =audit=, =plan=, and =upgrade-plan= use by avoiding the generic
plist/list-heavy Emacs compatibility path.  That did not remove the
main fixed costs:

- standalone NeLisp startup,
- source replay for stdlib + Nelix modules,
- manifest import evaluation,
- generic list/string/JSON work in NeLisp,
- allocator/GC pressure while building plist reports.

The runtime-image probe proved the boundary:

- explicit =NELIX_NELISP_IMAGE= is correct,
- replay-source images preserve =defun= during evaluation,
- replay-source images are slower than direct source load for Nelix.

Therefore the next speed lever is native/AOT execution, not automatic
runtime-image selection.

* Native Compilation Position

Native compilation is required for the real speed target, but it is not
sufficient by itself.

The current measurements split the problem into two layers:

- NeLisp native exec is now worthwhile.  The standalone
  =native-exec-elisp-artifact= path for the Nelix native subset dropped
  from ~24.6 sec to ~1.6 sec after bypassing manifest/plist validation
  and extracting the embedded object through the externless fast path.
- Nelix command latency is still dominated by runtime-side data work:
  manifest target matching, plist/list result construction, generic
  printer/JSON conversion, and allocator/GC pressure.

Therefore the correct direction is not "native-compile the whole Nelix
CLI".  It is to native-compile a small command-specific engine that:

1. receives compact line protocol or an equivalent cache-native format,
2. performs only name-set comparison,
3. emits compact JSON/name lines directly,
4. avoids building the Emacs-compatible report plist in the hot path.

Until that direct output path exists, native compilation can prove the
kernel but may not improve end-to-end =nelix --runtime nelisp audit= or
=upgrade-plan=.

* Goal

Create a command-specific native/AOT manifest engine for:

- =nelix --runtime nelisp list=
- =nelix --runtime nelisp audit MANIFEST=
- =nelix --runtime nelisp plan MANIFEST=
- =nelix --runtime nelisp apply MANIFEST --dry-run=
- =nelix --runtime nelisp upgrade-plan MANIFEST=

The engine must treat those commands as name-set operations over compact
line data, not as generic Emacs Lisp object traversal.

Target shape:

#+begin_src text
manifest source -> Nelix fast compiler -> AOT line protocol
profile names   -> newline names       -> AOT line protocol
                                      -> native command artifact
                                      -> minimal JSON or name lines
#+end_src

* Non-goals

- Do not native-compile the full Nelix CLI first.
- Do not depend on replay-source runtime images for speed.
- Do not require the Nix-free native store to be complete.
- Do not make the generic Emacs-compatible plist/list path faster before
  the command-specific path exists.
- Do not emit the full Emacs report structure from native code.

* AOT Input Protocol

The first committed boundary is =nelix-fast-aot-input= in
=nelix-fast.el=.  It exports a newline/tab protocol that a native
artifact can parse without reader, plist, alist, hash table, or JSON
dependencies.

Version 1:

#+begin_src text
NELIX-AOT-MANIFEST-V1
manifest<TAB>/abs/path/to/manifest.el
profile<TAB>default
system<TAB>x86_64-linux
target<TAB>magit<TAB>magit<TAB>legacyPackages.x86_64-linux.emacsPackages.magit
target<TAB>ripgrep<TAB>ripgrep
pin<TAB>ripgrep
installed<TAB>magit
installed<TAB>ripgrep
installed<TAB>fd
end
#+end_src

Rules:

- Fields are separated by literal TAB.
- Records are separated by literal LF.
- Package/profile fields must not contain TAB or LF.
- The native parser may reject unknown protocol versions.
- =target= record field 2 is the display target; fields 3..N are the
  acceptable installed-name candidates.
- =pin= records form a name set.
- =installed= records are profile names from the selected backend.

This is intentionally simpler than JSON.  JSON remains a CLI output
format, not the internal hot-path input format.

* Native Command Semantics

** list

=list= does not need manifest input.  The first native form may only
consume newline-separated profile names and write them back unchanged.
The useful speedup comes from avoiding generic formatting and keeping
the subprocess parsing path minimal.

** audit

Native audit is:

1. Build =installed-set= from =installed= records.
2. For each =target=, find the first installed candidate.
3. Emit =present= or =missing=.
4. Emit =extra= for installed names that are not in any target candidate
   set.

Skipped diagnostics remain explicit:

#+begin_src elisp
(:skipped (:state-pins :nelisp
           :lock-drift :nelisp
           :linux-command-audit :nelisp))
#+end_src

The native artifact may initially emit compact JSON directly:

#+begin_src json
{"ok":false,"present":["magit"],"missing":["ripgrep"],"extra":["fd"]}
#+end_src

The full Emacs path remains available through a later =--full= audit
mode.

** plan / apply --dry-run

Native plan and =apply --dry-run= are the same read-only convergence
calculation over the compact name protocol:

1. Build =installed-set= and =pin-set=.
2. For each =target=, find the first installed candidate.
3. If no installed candidate exists, add display target to =install=.
4. If an installed name is not declared by any target candidate, add it to
   =remove= unless pinned.
5. If an installed name is pinned but outside the manifest targets, report it
   as =protected=.

=plan= reports =status planned=.  =apply --dry-run= reports =status dry-run=.
Both paths must keep =dry-run true= and must not invoke Nix profile mutation.

** upgrade-plan

Native upgrade-plan is:

1. Build =installed-set= and =pin-set=.
2. For each =target=, find the installed candidate.
3. If no installed candidate exists, add display target to =missing=.
4. If installed name is pinned, add it to =pinned=.
5. Otherwise add installed name to =upgrade=.

The native artifact does not need to prove upstream freshness.  It
matches the current Nelix/Nix semantics: =nix profile upgrade= owns the
actual version decision.

* Artifact Strategy

Use NeLisp Doc 142 primitives in this order:

1. =.nelc= command artifact for a pure parser/formatter proof.
2. =.neln= native artifact for the compact parser and set comparison.
3. Optional standalone in-process native exec when the runtime loader can
   execute the needed builtin/boundary subset.

The first native target should not be a general loaded Nelix module.  It
should be a small command artifact with fixed entry points:

#+begin_src elisp
(nelix-aot-audit INPUT-TEXT)
(nelix-aot-upgrade-plan INPUT-TEXT)
(nelix-aot-list PROFILE-NAMES-TEXT)
#+end_src

The entry points should only use operations already practical in the
native/AOT subset:

- byte/string scan,
- integer indexes,
- equality over interned or copied strings,
- small hash/set equivalent or sorted-name comparison,
- direct output construction.

If hash table support remains too expensive for native exec, use sorted
name arrays:

1. parse names into vectors,
2. insertion sort or merge sort for ~200 entries,
3. binary search candidates.

At current scale, =O(n log n)= sorting over 200 names is acceptable and
may be simpler than native hash table support.

* Integration Plan

** M1: Stable AOT boundary

- =nelix-fast-aot-input= exports versioned input.
- Tests assert exact protocol records for aliases, pins, and installed
  names.
- Package fields containing TAB/LF are rejected.

Status: implemented in Doc 25 initial patch.

** M3: Native subset proof

- Restrict the parser to native-eligible forms.
- Compile to =.neln=.
- Use host-side native exec tests first.  Initial proof is shipped by
  =smoke-nelix-aot-native-artifact-host= for a minimal native-eligible
  defun and for the =nelix-aot-native-subset.el= set-comparison subset.
- Standalone real-runtime native command proof is available for the
  minimal native-eligible defun, a subset leaf kernel, and a composed
  subset call, plus string =str-len= / =str-byte-at= kernels over CLI
  arguments and byte-level line-protocol tag classification over a payload
  string.  Counted byte/tag scans over the payload are also proven through
  the extern-free recursive subset.  Field boundary detection and field
  byte-sum extraction are proven as the next parser step.  Package-name
  mask construction for =target=, =pin=, and =installed= rows is also
  proven through the same extern-free subset.  The public standalone CLI
  path uses a deliberately small composite proof artifact,
  =scripts/nelix-aot-native-cli-proof.el=, so one
  =native-exec-elisp-artifact= call proves string payload reads, tag
  offsets, field sums, and the sample mask code without linking the full
  host-only subset on every smoke invocation.  Target candidate field
  iteration is now proven on the host native subset: a target row with
  =magit= and =fd= candidates yields mask =5=.  Compact audit and
  upgrade-plan proof codes are also available as allocation-light numeric
  results.  Compact output selection and native-built line fragments are
  proven through the =.neln= string-result lane.  The next proof step also
  derives individual =present=, =missing=, and =upgrade= line fragments from
  payload masks instead of returning only fixed literal reports.

** M4: CLI integration

Original explicit environment gate:

#+begin_src sh
NELIX_NELISP_AOT=1 nelix --runtime nelisp --json audit MANIFEST
#+end_src

Current user-facing behavior: =nelix --runtime nelisp= enables the AOT
cache fast lane by default for supported manifest operations.  Set
=NELIX_NELISP_AOT=0=, =false=, =off=, =none=, or =direct= only when
debugging the slower direct standalone NeLisp path.

Status on 2026-06-18:

- =bin/nelix= wires AOT cache preference through direct NeLisp mode by
  injecting =nelix-fast-aot-force=, because standalone =getenv= does not
  currently see arbitrary process environment variables.
- =nelix-fast-audit= and =nelix-fast-upgrade-plan= can route small
  manifests through =scripts/nelix-aot-manifest-engine.el=.
- The portable line-protocol bridge is guarded by
  =nelix-fast-aot-portable-max-targets= and
  =nelix-fast-aot-portable-max-records=.  Large real profiles fall back
  to =nelix-fast= and report =:aot-skipped=.
- Real =/home/madblack-21/.emacs.d/nelix-env.el= has 199 targets.  It
  returns through the fallback in ~16.3 sec for =audit= and ~16.2 sec
  for =upgrade-plan= with
  =(:reason :portable-target-limit :targets 199 :record-limit 128 ...)=.
- A standalone payload-only probe showed the AOT engine can process a
  200-target prebuilt line protocol in ~0.2 sec.  The remaining slowdown
  is the Elisp bridge that imports the manifest and constructs the line
  protocol.  Therefore the next speed milestone is a cached/native parser
  artifact, not more work on the portable bridge.
- =nelix aot-cache MANIFEST= now writes a manifest-local target cache at
  =MANIFEST.nelix-aot-targets=.  The cache stores the protocol header,
  manifest/profile/system rows, target rows, and pin rows.  It
  intentionally omits installed profile rows and the final =end= record
  so runtime execution can append the current profile state.
- With the AOT cache fast lane enabled and an existing cache, =audit= and
  =upgrade-plan= can bypass manifest import and build the full AOT
  payload by concatenating the cache with =nix profile list= output.
  This is functionally correct for the real profile, but measured slower
  than the guarded direct fast path: ~19.0 sec for =audit= and ~17.9 sec
  for =upgrade-plan= versus ~16 sec fallback.  This confirms that the
  remaining bottleneck is result construction/output in NeLisp, not only
  manifest loading.
- =scripts/nelix-aot-manifest-engine.el= now provides direct JSON writers
  for =audit= and =upgrade-plan=.  =scripts/nelix-cli.el= can wrap this as
  raw JSON so =--json= output does not re-enter the generic
  =nelix-cli--json-encode= plist/list printer.  ERT coverage locks the
  direct writer and CLI pass-through behavior.
- Direct JSON improves the real cached route but does not reach the M5
  target yet.  On the 199-target real manifest, with the cache generated
  outside the hot path, =audit= measured ~18.2 sec and =upgrade-plan=
  measured ~16.8 sec.  The remaining hot work is therefore the
  line-protocol parse, hash/list construction inside the portable engine,
  and runtime construction of =installed= records from =nix profile list=.
- Generating =aot-cache= under standalone NeLisp is not a hot-path goal.
  It imports the real manifest and can take too long; cache generation
  should remain an Emacs/host maintenance operation until a manifest
  compiler/cache format exists.
- A =bin/nelix= fast lane now detects
  =NELIX_RUNTIME=nelisp [--json] audit|plan|apply --dry-run|upgrade-plan MANIFEST=
  when AOT cache use is enabled.  In that case it builds or refreshes
  =MANIFEST.nelix-aot-targets= when needed, builds the runtime payload in
  the shell, and runs only
  =scripts/nelix-aot-manifest-engine.el= under =target/nelisp= instead of
  loading the full Nelix CLI/bootstrap suite.  JSON mode uses the direct
  JSON writers; non-JSON mode uses compact tab-separated line writers
  (=nelix-aot-audit-lines= / =nelix-aot-upgrade-plan-lines=), so the
  cached fast lane does not re-enter the generic plist printer either.
  This keeps the direct NeLisp fallback available through
  =NELIX_NELISP_AOT=0= while removing several seconds of CLI fixed cost
  from the default standalone runtime.  The same shell fast lane is
  now covered for ID-only runtime payloads: fake-profile smokes exercise both
  =audit= and =upgrade-plan= with =target-id= / =pin-id= records and no
  runtime string =target= / =pin= records, in both compact line output and
  =--json= direct-writer mode.  =make smoke-nelix-aot-cache-fast-lane=
  makes that =bin/nelix= coverage repeatable without touching the user's real
  Nix profile.
- Fast-lane real-profile measurements on 2026-06-18:

| Step | Measurement |
|------+-------------|
| cache generation through Emacs runtime | ~0.066 sec |
| cached =audit= through fast lane | ~13.96 sec |
| cached =upgrade-plan= through fast lane | ~12.73 sec |

- A hash-free linear comparison experiment was measured and rejected.
  It exceeded 55 sec on the real audit payload, while the existing
  hash-index JSON engine took ~13.94 sec for the same payload.  The next
  real speed lever is therefore not linear Elisp comparison; it is moving
  the parser/index builder to a native artifact or a lower-level
  cache-native format.
- NeLisp native exec now has a host-proof string argument lane for
  =.neln= general native execution.  The C harness can construct
  =Sexp::Str= slots from command-line strings and pass their slot
  pointers to native defuns.  =scripts/nelix-aot-native-subset.el= now
  proves =str-len= and =str-byte-at= over a string argument in
  =smoke-nelix-aot-native-artifact-host=, returning =3= and =97= for
  ="abc"=.  This does not yet make the full line-protocol parser native,
  but it removes the previous blocker where native proofs could only
  accept integer arguments.

** M5: Automatic selection

Enable automatic AOT only after measurements prove it is faster than the
direct fast path on:

- =list=
- =audit ~/.emacs.d/nelix-env.el=
- =upgrade-plan ~/.emacs.d/nelix-env.el=

Do not auto-select replay-source runtime-image mode for speed.

* Open Issues

- NeLisp native exec still has limited general builtin/boundary coverage.
- =target/nelisp= now routes the public =nelisp--syscall-read-file=
  compatibility name through =rdf=, so general evaluation and artifact
  loading can use the normal file reader without the earlier process-backed
  =cat= workaround.  The older long-name native dispatch path should still
  be audited later.
- Compiling the full =scripts/nelix-aot-manifest-engine.el= source as
  canonical =.nelc= is still too slow for a smoke gate.  The current
  artifact smoke compiles a tiny canonical artifact and keeps the full
  engine validation in =smoke-nelix-aot-engine-nelisp=.
- Loading the full engine from inside =eval-elisp-artifact= still exposes
  NeLisp's incomplete =require= / feature-state behavior.  The full AOT
  engine artifact remains an M3/M4 target after that runtime gap is fixed.
- M3 has a real =.neln= native artifact proof for a minimal
  native-eligible defun through both host and standalone command paths,
  plus host and standalone proofs for the bitset-based Nelix name-set
  subset.  The previous standalone hang was caused by routing the embedded
  object payload through generic reader/string processing; native exec now
  uses a fast object extraction path, and the public CLI has an externless
  fast path that skips manifest/plist validation before falling back to the
  validated path.  The remaining native gap is reducing the ~1.6 second
  standalone/toolchain fixed cost, extending the proof to the full
  line-protocol manifest engine subset, and eventually replacing the
  host-toolchain harness with in-process mmap/relocation.
- Host-side and standalone CLI native exec can now pass string arguments to
  native defuns, which unblocks parser-kernel proofs over payload text.
  The remaining bridge is not public argument parsing; it is lowering the
  full line-protocol parser into the native subset and reducing the
  standalone/toolchain fixed cost.
- The native subset now includes the first byte-level line-protocol parser
  kernel.  =nelix-aot-native-protocol-prefix-code= validates the
  =NELIX-AOT-MANIFEST-V1= prefix, and
  =nelix-aot-native-record-tag-code= classifies record tags at explicit
  offsets (=target=1, =pin=2, =installed=3, =end=4).  The
  =smoke-nelix-aot-native-artifact-host= gate proves both host native exec
  and public =target/nelisp native-exec-elisp-artifact= over a real
  line-protocol payload string.
- The native subset also includes counted payload scans without mutable
  locals: =nelix-aot-native-count-byte= counts bytes, and
  =nelix-aot-native-count-tag= scans the payload for record tags.  These
  use tail recursion and inline =(+ x 1)= arithmetic, which keeps the
  compiled =.neln= artifact extern-free and avoids the current
  =while= / =setq= lowering gap.
- Field extraction has started in the same extern-free subset.
  =nelix-aot-native-find-byte= finds delimiter offsets,
  =nelix-aot-native-field-end= finds TAB/LF/EOF field boundaries, and
  =nelix-aot-native-field-byte-sum= proves stable field content reads
  (=magit= -> =530=, =ripgrep= -> =761= in the smoke payload).  The
  standalone proof uses =nelix-aot-native-line-proof-code= to combine
  version/tag/count/field checks into one public native exec invocation so
  the correctness gate does not multiply the standalone/toolchain fixed
  cost unnecessarily.
- Package-name mask construction is now in the native subset.
  =nelix-aot-native-name-bit-from-sum= maps the proof package names to
  8-bit set bits, =nelix-aot-native-mask-for-tag= scans the line protocol
  for selected tag rows, and the =desired=, =pin=, and =installed= masks
  are built without plist/list/hash table allocation.  The sample payload
  proves =desired=1=, =installed=1=, =pin=2=, and mask equality =1=, with
  host-side =nelix-aot-native-mask-proof-code= returning =19=.  The
  standalone public CLI artifact keeps only a compact payload proof and
  returns =556=, combining byte length, tag offsets, field sum, and the
  same sample mask code.
- Target candidate field iteration is proven in the host native subset.
  =nelix-aot-native-target-candidate-mask-at= skips the first target
  payload field and scans candidate fields until LF; the multi-candidate
  proof payload =target\tmagit\tmagit\tfd= returns candidate mask =5=.
  =nelix-aot-native-compact-audit-code= returns =9= for the base payload,
  and =nelix-aot-native-compact-upgrade-plan-code= returns =10= for the
  candidate payload.  Portable compact line/name output now exists in
  =scripts/nelix-aot-manifest-engine.el= and is wired through the shell
  AOT cache fast lane for non-JSON =audit= and =upgrade-plan=.  The next
  native step has started: =nelix-aot-native-compact-audit-output-select=
  and =nelix-aot-native-compact-upgrade-output-select= prove that the
  native parser can select and return compact output strings through the
  =.neln= string-result lane.  The public standalone CLI proof also
  verifies native string-result return.  Native-built literal line
  fragments are now proven by =nelix-aot-native-compact-audit-lines-proof=,
  =nelix-aot-native-compact-upgrade-lines-proof=, and
  =nelix-aot-native-cli-lines-proof=.  Field-derived line fragments are
  also proven on the host native subset: =present\tmagit=,
  =missing\tfd=, and =upgrade\tmagit= are selected from payload-derived
  desired/installed/pin/candidate masks, not from a fixed report literal.
  The host subset now also proves small multi-line compact reports:
  =ok\tfalse= with =present\tmagit= and =missing\tfd= for audit, and
  =operation\tupgrade= with =upgrade\tmagit= and =missing\tfd= for
  upgrade-plan.  This deliberately avoids runtime =concat= for now:
  =concat= is still treated as a delegated builtin by the native compiler,
  while branch-selected line/report literals stay in the lightweight
  =nl_alloc_str= lane.  The native-safe builder lane has now started:
  the host =.neln= proof harness supports =nl_alloc_mut_str=,
  =nl_mut_str_push_byte=, and =nl_mut_str_finalize=, and
  =nelix-aot-native-builder-audit-report-proof= builds the same compact
  audit report byte-by-byte in a caller-provided output slot after checking
  payload-derived masks.  The first reusable append helpers are also
  proven: =nelix-aot-native-builder-push-ok-false=,
  =nelix-aot-native-builder-push-present-magit=,
  =nelix-aot-native-builder-push-missing-fd=, and
  =nelix-aot-native-builder-push-backend-nix= can be called from the
  report builder without leaving the native-safe integer/string slot lane.
  Bit-selected append dispatch is now proven for the proof name subset:
