* STATUS

SHIPPED 2026-06-08.  Phase 5 groundwork on canonical =main=.

Phase 4-C introduced =anvil-pkg-compat= as the runtime boundary but
left NeLisp async / HTTP as unsupported branches.  This Phase 5
groundwork closes the integration gap without changing the public
=pkg-= API: call sites keep using compat primitives, while NeLisp can
provide native process / network backends as its Layer-2 packages
become available.

* Goal

| Before Phase 5 groundwork                                      | After Phase 5 groundwork                                           |
|----------------------------------------------------------------+---------------------------------------------------------------------|
| NeLisp runtime detection depended on =nelisp-call-process= only | Any usable split backend (=call-process= / async / HTTP) is enough  |
| Async and HTTP required future edits at call sites              | Compat hook vars let NeLisp plug in implementations                 |
| Loaded NeLisp process / HTTP backends were ignored              | Loaded backends are auto-detected when hooks are unset              |
| HTTP fallback had only the Emacs =url= backend                  | NeLisp can use native backends or =curl= fallback for text / bytes   |
| State JSON wrote through Emacs =json-serialize= directly        | State writes go through =anvil-pkg-compat-json-serialize=           |
| File / PATH helpers assumed loaded Emacs-compat functions       | Helpers lazy-require NeLisp runtime / Emacs-compat / fileio packages |
| Async sentinels used raw Emacs process property/status APIs     | Async process metadata / status / exit access goes through compat   |
| Async stderr buffers used raw Emacs buffer APIs                  | Buffer create / read / live / kill helpers go through compat        |

* Locked decisions

** L51 — hooks first, backend probe second, auto-detect third

The NeLisp branch checks explicit hooks before auto-detection:

- =anvil-pkg-compat-nelisp-make-process-function=
- =anvil-pkg-compat-nelisp-call-process-function=
- =anvil-pkg-compat-nelisp-getenv-function=
- =anvil-pkg-compat-nelisp-executable-find-function=
- =anvil-pkg-compat-nelisp-http-get-function=
- =anvil-pkg-compat-nelisp-http-get-binary-function=

Hooks are the stable integration point for NeLisp while its Layer-2
APIs are still moving.  When a hook is unset, compat probes
=nelisp-process= / =nelisp-network= / =nelisp-http= with noerror
require once, then auto-detects =nelisp-make-process=,
=nelisp-http-get=, =nelisp-http-fetch=, and
=nelisp-http-get-binary= if they are available.  This keeps the
mainline package useful with both explicit embedding and package-split
NeLisp images.

** L52 — lazy runtime refresh

Runtime detection starts at load time, but =anvil-pkg-compat-runtime=
refreshes a stale nil result.  The detection probe also attempts
noerror requires for =nelisp-process=, =nelisp-network=, and
=nelisp-http= once.

Subprocess dispatch must consult =anvil-pkg-compat-runtime=, not read
=anvil-pkg-compat--nelisp-runtime-p= directly.  This prevents a session
that loaded =anvil-pkg= before NeLisp backends from staying stuck on
the Emacs branch after those backends appear.  File I/O is slightly
more permissive: it probes =nelisp-runtime=,
=nelisp-emacs-compat=, and =nelisp-emacs-compat-fileio= once, prefers
NeLisp Layer-2 file primitives when the runtime is NeLisp, and still
allows Emacs-compatible file primitives as a bootstrap / test fallback
when no NeLisp fileio package is loaded.

Async process sentinels store and read their metadata through
=anvil-pkg-compat-process-get= / =anvil-pkg-compat-process-put= and
query status / exit code through compat wrappers.  This keeps the
Emacs process path unchanged while allowing future native
=nelisp-process= wraps to use their own property and status APIs.

Async stderr buffers are created, inspected, and cleaned up through
=anvil-pkg-compat-generate-buffer=,
=anvil-pkg-compat-buffer-string=,
=anvil-pkg-compat-buffer-live-p=, and
=anvil-pkg-compat-kill-buffer=.  NeLisp can route those operations via
=nelisp-ec-*= without requiring async call sites to use raw Emacs
buffer functions.

** L53 — binary HTTP native backend, then =curl=

Native binary HTTP is still preferred via
=anvil-pkg-compat-nelisp-http-get-binary-function=.  When no explicit
hook is installed, the NeLisp branch auto-detects
=nelisp-http-get-binary=.  If neither native path exists, it uses
=curl= when present on PATH.

Text HTTP follows the same fallback policy after checking
=anvil-pkg-compat-nelisp-http-get-function= and a loaded
=nelisp-http-get= backend, then the higher-level
=nelisp-http-fetch= package.  The text adapter reuses the binary curl
path and returns only =:status= and =:body= to preserve the existing
contract.

Auto-detected native HTTP backends are treated as preferred but not
infallible.  If a loaded =nelisp-http-get= / =nelisp-http-fetch= /
=nelisp-http-get-binary= path normalizes to status 0 because the
standalone image lacks its lower runtime primitives, compat retries
through =curl= when available.  If =curl= is unavailable, the original
status 0 response is preserved.

The fallback writes headers and body to separate temp files using:

#+begin_src shell
curl -L --silent --show-error --max-time TIMEOUT \
  --dump-header HEADERS --output BODY [-H "Authorization: ..."] URL
#+end_src

The final redirect response block determines =:status= and
=:content-length=.  If =curl= is absent, compat preserves the existing
=anvil-pkg-http-not-supported= signal.

** L54 — JSON serialization is a compat primitive

=anvil-pkg-state= no longer calls =json-serialize= directly.  It writes
through =anvil-pkg-compat-json-serialize=, which prefers Emacs
=json-serialize= and falls back to =nelisp-json-serialize=.

State entries store missing =expires-at= as nil.  That keeps the JSON
shape portable because NeLisp JSON encoders can represent nil as
=null= without needing Emacs's =:null= sentinel object.

* Remaining external completion blockers

** Ownership boundary

These blockers are intentionally classified as NeLisp runtime /
Layer-2 backend work, not anvil-pkg application work.

anvil-pkg owns:

- Stable compat entry points and hook variables:
  =anvil-pkg-compat-nelisp-call-process-function=,
  =anvil-pkg-compat-nelisp-make-process-function=,
  =anvil-pkg-compat-nelisp-http-get-function=, and
  =anvil-pkg-compat-nelisp-http-get-binary-function=.
- Result-shape validation for those hooks before anvil-pkg consumes
  the returned plists.
- Lazy detection of loaded NeLisp package functions such as
  =nelisp-make-process=, =nelisp-http-get=, and
  =nelisp-http-fetch=.
- Graceful unsupported-runtime signals when no executable backend is
  available.
- Opportunistic =curl= fallback for HTTP when a process backend can run
  =curl=.
- Smoke/readiness tests that report whether the current standalone
  image can execute the full anvil-pkg suite.

NeLisp owns:

- The executable process lower primitive: either a standalone
  =make-process= equivalent or a native =nelisp-make-process= path
  backed by syscall / FFI primitives such as spawn, pipe, and wait.
- The executable HTTP lower primitive: either a standalone
  =url-retrieve-synchronously= equivalent or a native
  =nelisp-http-get= / =nelisp-http-fetch= transport that does not
  depend on Emacs's host URL library.
- Runtime semantics for process objects, status, exit code, buffers,
  and network I/O that NeLisp packages expose upward.

The boundary is deliberately narrow: anvil-pkg must not grow its own
Rust syscall bridge, process scheduler, socket/TLS stack, or host-URL
implementation.  Doing so would duplicate NeLisp runtime ownership and
would not help other NeLisp packages.  Conversely, NeLisp does not need
to know anvil-pkg's install DSL; it only needs to provide executable
process / HTTP primitives matching the small compat contracts above.

- [ ] Replace the async hook with a production NeLisp native backend
      backed by an executable lower process primitive.  The current
      local NeLisp checkout has =nelisp-make-process= /
      =nelisp-call-process= definitions in =nelisp-process.el=, but
      until the syscall FFI bridge is wired those definitions fall
      through to host =make-process=.  The standalone image used by
      the smoke target currently reports no lower =make-process=
      primitive, so the loaded definitions are useful for
      auto-detection / contract probing but not yet executable.  This
      blocker is cleared by either routing =nelisp-make-process=
      through the native =nl-syscall-posix-spawn= /
      =nl-syscall-waitpid= / =nl-syscall-pipe2= path, or by exposing a
      standalone =make-process= equivalent that can run subprocesses.
- [ ] Replace the text HTTP hook with a production NeLisp native
      backend backed by an executable lower HTTP primitive.  The
      current local NeLisp =nelisp-network.el= and =nelisp-http.el=
      sources provide =nelisp-http-get= / =nelisp-http-fetch=
      definitions, but those paths delegate to host
      =url-retrieve-synchronously=.  The standalone image used by the
      smoke target currently reports no lower
      =url-retrieve-synchronously= primitive, so the definitions are
      loadable and visible to anvil-pkg compat probing but not yet a
      production standalone HTTP transport.  This blocker is cleared by
      adding a standalone URL / HTTP primitive or by routing
      =nelisp-http-fetch= through a native transport that does not
      depend on the host URL library.
- [ ] Run the suite under a real NeLisp standalone image.  The local
      ERT shim, =cl-letf= support, runtime-mode test registration, and
      per-file loadability are already green for all 182 tests.  Full
      execution is intentionally held behind the two native lower
      primitive blockers above so =make smoke-nelisp-suite= cannot
      report a misleading pass on a runtime that can load definitions
      but cannot execute process / HTTP behavior.

* Out of scope

- Changing the public =pkg-= API.
- Reworking the Emacs =url= backend.
- Persisting credentials.  Phase 4-G's env-var-only credential model
  remains unchanged.
- Removing unsupported-runtime signals.  They remain the correct result
  when no native backend and no fallback exists.
