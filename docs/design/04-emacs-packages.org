* STATUS

Phase 4-A — DESIGN.  Implementation will be dispatched to codex via
anvil-orchestrator (3 parallel sub-tasks); claude judges + integrates.

This phase adds an /Emacs-package/ build-system to anvil-pkg's DSL so
Elisp libraries (magit, helm, transient, ...) can be installed by the
same =pkg-define= / =pkg-install= surface that already covers system
tools.  The Nix builder is =pkgs.emacsPackages.trivialBuild= /
=melpaBuild=, so reproducibility / binary cache / load-path
augmentation all come for free.

* Goal

Replace the user's =async-installer= flow:

#+begin_src elisp
;; Old (async-installer):
(async-installer-git-add "magit/magit"
                         :commit "v3.3.0")

;; New (anvil-pkg Phase 4-A):
(pkg-define magit
  (version "3.3.0")
  (source (github-fetch :owner "magit" :repo "magit"
                        :rev "v3.3.0"
                        :sha256 "sha256-..."))
  (build-system emacs-package))
(pkg-install 'magit)   ; -> Nix builds, load-path augmented, (require 'magit) works.
#+end_src

* Non-goals

- =melpaBuild= dependency-graph resolution.  The user supplies
  =:depends-on= manually; we do not introspect =Package-Requires=.
  Phase 4-B will scrape the header.
- Pure-Emacs (no Nix) install path.  Phase 4-A requires Nix; the
  legacy =async-installer= continues to cover Nix-less environments.
- Mutating the user's running =EMACSLOADPATH= permanently.  We add
  the store path to =load-path= /in-process/; the user's init.el can
  re-add the same path on next startup if they want persistence.

* Locked decisions

** L7. Build-system name = =emacs-package=

Plain =emacs-package=, not =elisp= or =melpa= or =trivial-build=.
Reason: matches Elisp users' mental model — "this is an Emacs Lisp
package" — and stays neutral on the Nix builder underneath
(=trivialBuild= for git-fetch'd raw .el dirs, =melpaBuild= for
recipe-style packaging).  Phase 4-A picks =trivialBuild=; Phase 4-B
adds the =:format= switch for =melpaBuild=.

** L8. DSL surface

Symbol form (defaults to trivialBuild, no extra args):

#+begin_src elisp
(build-system emacs-package)
#+end_src

Args form (Phase 4-B will recognise =:format=):

#+begin_src elisp
(build-system (emacs-package :format "trivial"))
#+end_src

Plus a new top-level sub-form keyword =:depends-on= for declaring
Elisp dependencies that are themselves anvil-pkg-installed:

#+begin_src elisp
(pkg-define magit
  ...
  (build-system emacs-package)
  (depends-on (list dash transient with-editor)))
#+end_src

These map to =propagatedBuildInputs = with pkgs.emacsPackages; [...]=
in the rendered Nix derivation, so building magit pulls in dash,
transient, with-editor as Emacs packages and adds them to its
load-path.

** L9. Render target = =pkgs.emacsPackages.trivialBuild=

The simplest of the nixpkgs Emacs builders.  Takes =pname= /
=version= / =src= / =packageRequires= (= propagated inputs).  Output
is a derivation whose =share/emacs/site-lisp/<pname>/= is the
.el(c) install directory.

Phase 4-A renderer skeleton:

#+begin_src nix
pkgs.emacsPackages.trivialBuild {
  pname = "magit";
  version = "3.3.0";
  src = pkgs.fetchFromGitHub {
    owner = "magit"; repo = "magit"; rev = "v3.3.0";
    sha256 = "sha256-...";
  };
  packageRequires = with pkgs.emacsPackages; [ dash transient with-editor ];
}
#+end_src

** L10. Post-install load-path augmentation

After =(pkg-install 'pkg)= for an =emacs-package= entry, the new
helper =anvil-pkg--emacs-package-after-install= must:

1. Resolve the store path of the installed package (from =nix profile
   list --json='s =storePaths= field).
2. Locate =share/emacs/site-lisp/<pname>/= within that store path.
3. =add-to-list 'load-path= for that directory.
4. Optionally =require= the package — only when the caller passed
   =:require= in the install call.  No automatic require by default
   (let the user choose when to load).

Implementation lives in =anvil-pkg.el= since the post-install hook
needs access to the Phase 1 install result plist.  NeLisp standalone
can re-use the same code unchanged — =add-to-list= is already core,
=require= is already in the Rust evaluator's builtin set.

** L11. Default profile dir for emacs-package installs

Same =anvil-pkg-profile-dir= as system-tool installs.  Sharing the
profile keeps a single =nix profile list= showing both kinds — and
the load-path augmentation already reads the profile JSON.  No
separate dir.

** L12. async-installer migration story

Document only — do NOT bundle a converter in this phase.  The user
rewrites their async-installer entries to =pkg-define= forms
manually; later Phase 4-B may ship a =anvil-pkg-import-async-installer=
utility that scans an =async-installer-git-list= variable and emits
=pkg-define= forms.

* Open questions

** OQ4. =:require= timing relative to flake build

If =pkg-install= returns =t= while the build is still in flight (it
shouldn't with synchronous =call-process= but worth checking on slow
builds), the =:require= would fire on a non-existent file.  Phase
4-A relies on synchronous nix; Phase 4-B may make it async and need
a sentinel chain like async-installer's.

Tentative: synchronous in 4-A, defer async to Phase 4-B.

** OQ5. melpaBuild vs trivialBuild

trivialBuild is the simplest builder but does not apply Elisp
package conventions like =Package-Requires= header parsing or
=byte-compile-file= invocation.  melpaBuild does.  For Phase 4-A
trivialBuild is enough; Phase 4-B can add =:format "melpa"=.

** OQ6. Native compilation

nixpkgs's trivialBuild can be wrapped with
=emacsPackages.withNativeCompilation = true=.  Phase 4-A defaults to
unwrapped (= no native comp); Phase 4-B can flip the default once
real-world performance data is in.
