#+TITLE: Doc 27 - Nelix init.el Migration Workflow
#+DATE: 2026-06-19
#+STATUS: ACTIVE

* 1. Purpose

This document defines the practical workflow for migrating a real user
=init.el= package setup to Nelix =lock / plan / apply=.

The target state is:

- Nelix itself is installed from a Debian package;
- the user's desired package state lives in one manifest entry point;
- Emacs package names and Linux/Nix tools are read from lightweight index files;
- =nelix lock= records the resolved package state;
- =nelix plan= and =nelix apply --dry-run= show the profile diff before any
  mutation;
- real =apply= is only run after reviewing install/remove commands.

This stage remains Nix-backend first.  The native store design is separate.

* 2. Standard User Manifest

For init migration, the standard user entry point is:

#+begin_example
~/.emacs.d/nelix-package.el
#+end_example

The file must be state-only.  It should not configure editor behavior, define
keybindings, start timers, install packages on load, or require full init
modules.  The current personal manifest shape is:

#+begin_src emacs-lisp
;;; nelix-package.el --- Personal Nelix desired-state manifest -*- lexical-binding: t; -*-

(require 'nelix)

(defvar nelix-package-emacs-packages)
(defvar nelix-linux-base-nix-packages)
(defvar nelix-linux-bootstrap-apt-packages)

(defconst nelix-package-user-emacs-directory
  "/home/madblack-21/.emacs.d/")

(load (expand-file-name "custom-lisp/nelix-package-index.el"
                        nelix-package-user-emacs-directory)
      nil nil t)
(load (expand-file-name "custom-lisp/nelix-linux.el"
                        nelix-package-user-emacs-directory)
      nil nil t)

(nelix-environment
 (name "madblack-21")
 (profile "default")
 (nix-channel "nixpkgs")
 (imports "custom-lisp/nelix-package-index.el"
          "custom-lisp/nelix-linux.el")
 (backend-policy
  (gnu/linux nix nelix-native)
  (darwin nix nelix-native)
  (windows-nt nix))
 (emacs-packages nelix-package-emacs-packages)
 (linux-packages nelix-linux-base-nix-packages)
 (bootstrap-apt-packages nelix-linux-bootstrap-apt-packages))
#+end_src

The explicit =nelix-package-user-emacs-directory= is intentional for now:
standalone NeLisp validates this form reliably, while =user-emacs-directory=
is not yet a stable runtime assumption.

The verifier requires exactly one top-level =nelix-environment= form and rejects
returning to a top-level =nelix-manifest= form for this real user manifest.

* 3. init.el Extraction Rule

The migration does not copy the whole =init.el=.  It extracts only package
identity into generated state files:

- =~/.emacs.d/custom-lisp/nelix-package-index.el= contains the lightweight
  Emacs package name list and Nix attribute aliases;
- =~/.emacs.d/custom-lisp/nelix-linux.el= contains Linux/Nix package groups and
  bootstrap APT anchors;
- the old full recipe file can remain separate for manual recipe work, but it
  is not the default manifest dependency.

Extraction should scan only package declaration/install surfaces, such as
=package-install=, =use-package= package names, generated recipe names, and
Linux tool groups.  It must not import arbitrary init side effects.

* 4. Required Commands

The basic real-user flow is:

#+begin_src sh
manifest="$HOME/.emacs.d/nelix-package.el"

make verify-user-manifest-dsl
nelix --runtime nelisp --json validate "$manifest"
nelix --json lock "$manifest"
nelix --json plan "$manifest"
nelix --json apply "$manifest" --dry-run
#+end_src

=make verify-user-manifest-dsl= is the source-tree regression guard for this
real manifest.  It checks the top-level DSL shape and validates through normal
Nelix.  Its default auto mode also attempts standalone NeLisp validation when
the runtime is available.  The =nelix --runtime nelisp --json validate=
wrapper keeps the standard manifest off the slow full standalone NeLisp
manifest load path.  The CLI exposes =nelix-fast-validate-json= as the
source-only validate API, and the wrapper may delegate that validation to Emacs
batch until the native reader/count scanner is fast enough for the 200-package
manifest.

=NELIX_USER_MANIFEST_NELISP=required= also requires the read-only AOT cache lane
for =audit=, =plan=, =apply --dry-run=, and =upgrade-plan=.  Those commands
must report =:nelisp-aot-cache= as the fallback and must not enter the slow full
standalone manifest path.  =lock= may still run through Emacs batch until the
NeLisp process/runtime path is complete enough for mutating lock writes.
The verifier prints the =audit= missing/extra counts and names plus the
=apply --dry-run= remove candidate count and names.
=NELIX_USER_MANIFEST_MAX_MISSING=N=,
=NELIX_USER_MANIFEST_MAX_EXTRA=N=, and
=NELIX_USER_MANIFEST_MAX_REMOVE=N= can make the gate fail when the real profile
diff is larger than the operator has explicitly allowed.
It also compares =nelix --runtime nelisp --json plan MANIFEST --dry-run= with
=nelix --runtime nelisp --json apply MANIFEST --dry-run= across =install=,
=remove=, =keep=, =protected=, and =commands= so the review command and dry-run
apply cannot drift silently.

=NELIX_USER_MANIFEST_LOCKED=required= strengthens the same source-tree verifier
with a real manifest lock cycle: it backs up any existing
=MANIFEST.nelix-lock=, runs =nelix --json lock MANIFEST=, verifies
=nelix --json apply MANIFEST --locked --dry-run=, and restores the previous
lock file.  When =NELIX_USER_MANIFEST_NELISP=required= is also set, it verifies
the same locked dry-run through the NeLisp AOT cache fast lane and requires the
result to report =:nelisp-aot-cache=.  This covers the daily locked review path
without mutating the real profile.

The lock file path is:

#+begin_example
~/.emacs.d/nelix-package.el.nelix-lock
#+end_example

The lock records manifest/import digests and per-package target resolution.

* 5. Plan Review

=nelix plan= is read-only.  =nelix apply --dry-run= returns the same action
model with =:status dry-run= and does not mutate the profile.

Before real =apply=, review at least:

- =install= count and command rows;
- =remove= count and command rows;
- =protected= rows from pins;
- whether transitional wrapper packages should be kept in =:linux= or removed;
- whether any one-off profile tools should be added to the manifest.

On 2026-06-19, the current personal profile dry-run result was:

#+begin_example
install=0
remove=0
keep=203
protected=0
commands=0
#+end_example

The earlier remove candidates were kept explicitly in
=~/.emacs.d/custom-lisp/nelix-linux.el=:

nelix-system-dh-elpa     -> transitional Debian tool wrapper

#+end_example

This keeps the profile converged without removing currently useful tools.

* 6. Daily Init Migration Verification

The package release gates intentionally do not load the user's
=~/.emacs.d/init.el= or =~/.emacs.d/early-init.el=.  That keeps
=deb-release-gate= and =deb-full-gate= suitable for package verification and
prevents root-owned or daemon-oriented user init side effects.

The explicit daily-use verifier is:

#+begin_src sh
NELIX_USER_MANIFEST="$HOME/.emacs.d/nelix-package.el" \
NELISP=/home/madblack-21/Cowork/Notes/dev/nelisp/target/nelisp \
NELISP_ROOT=/home/madblack-21/Cowork/Notes/dev/nelisp \
make verify-user-init-migration
#+end_src

It performs two checks in order:

- =nelix --runtime nelisp= read-only manifest commands:
  =validate=, =audit=, =plan=, =apply --dry-run=, =upgrade-plan=, and locked
  =apply --dry-run=;
- direct comparison of the init-migration =plan --dry-run= and
  =apply --dry-run= action sets;
- Emacs batch loading of =~/.emacs.d/early-init.el= and =~/.emacs.d/init.el=,
  followed by =my-nelix-audit=.

This target is the current evidence that the extracted package manifest and the
real editor startup path agree.  It can run against the source tree without
requiring the installed Debian package to be current.  The Debian package
version check is enabled automatically only when the verifier is using the
installed =/usr/bin/nelix= or installed ELPA source directory; set
=NELIX_INIT_MIGRATION_REQUIRE_DEB=required= to force it or =skip= to disable it
explicitly.  The standard audit mode is =NELIX_INIT_MIGRATION_AUDIT=required=,
which runs =my-nelix-audit= after loading init.  During source-tree development,
=NELIX_INIT_MIGRATION_AUDIT=load-only= verifies that the real init path loads
and defines =my-nelix-audit= without requiring the installed Debian bootstrap
package to already match the source-tree version.

The init migration verifier also prints and can cap the real profile diff before
loading init.  Set =NELIX_INIT_MIGRATION_MAX_MISSING=N=,
=NELIX_INIT_MIGRATION_MAX_EXTRA=N=, and
=NELIX_INIT_MIGRATION_MAX_REMOVE=N= to enforce init-specific limits; when unset,
the corresponding =NELIX_USER_MANIFEST_MAX_*= values are reused.

* 7. Debian Package Verification

Installed package verification requires =elpa-nelix >= 0.1.0-4= because older
packages did not ship =/usr/bin/nelix=.

After installing the rebuilt package:

#+begin_src sh
sudo apt install --reinstall ../elpa-nelix_0.1.0-4_all.deb

NELIX_USER_MANIFEST="$HOME/.emacs.d/nelix-package.el" \
NELIX_USER_MANIFEST_NELISP=1 \
NELIX_USER_MANIFEST_LOCKED=required \
NELISP=/home/madblack-21/Cowork/Notes/dev/nelisp/target/nelisp \
NELISP_ROOT=/home/madblack-21/Cowork/Notes/dev/nelisp \
packaging/verify-installed-nelix-debian.sh 0.1.0-4
#+end_src

Without root access, the package payload can still be verified by extraction:

#+begin_src sh
NELIX_USER_MANIFEST="$HOME/.emacs.d/nelix-package.el" \
NELIX_USER_MANIFEST_NELISP=1 \
NELIX_USER_MANIFEST_LOCKED=required \
NELISP=/home/madblack-21/Cowork/Notes/dev/nelisp/target/nelisp \
NELISP_ROOT=/home/madblack-21/Cowork/Notes/dev/nelisp \
packaging/verify-extracted-nelix-debian.sh ../elpa-nelix_0.1.0-4_all.deb 0.1.0-4
#+end_src

* 8. Open Edges

- =nelix --runtime nelisp --json validate= works for the standard manifest
  without entering the full standalone NeLisp manifest loader, and the read-only
  =audit / plan / apply --dry-run / upgrade-plan= path is covered by the AOT
  cache fast lane.  The standalone NeLisp Nix apply path now supports
  transaction rollback and =apply --locked --dry-run= without loading the lock
  as executable Elisp.  The installed package gate now covers
  =apply --locked --allow-remove-count 1= against a fake Nix profile mutation.
  A final real-user profile run with an actual Nix profile is still a release
  acceptance check rather than a unit/source-tree invariant.
- =apply= converges the profile and can remove unmanaged entries.  This is
  intentional, but docs and UI must keep the review step prominent.
- Transitional Debian wrapper packages need a final policy: either keep them in
  =:linux= while they are still expected, or let =apply= remove them.
- The native store is not part of this workflow yet.
