#+TITLE: Doc 28 - Nelix Apply Transaction, Debian Gates, and DSL v1
#+DATE: 2026-06-19
#+STATUS: ACTIVE

* 1. Purpose

This document fixes the next practical safety layer after the real
=init.el= migration flow:

- =nelix apply= must be recoverable when a Nix command fails halfway through;
- Debian package gates must exercise the installed =/usr/bin/nelix= CLI, not
  only source-tree helpers;
- the manifest DSL needs a stable v1 contract before native store and lockfile
  schema work expand the surface.

* 2. Apply Transaction Model

For the Nix backend, =nelix apply= treats profile mutation as a transaction
bounded by the active Nix profile generation.  The same transaction record
shape now also applies to =nelix-native= real apply: the rollback boundary is
the native profile generation recorded under =nelix-profile-root=.

Default behavior:

1. Build the normal plan.
2. Before executing install/remove commands, call =pkg-list-generations= and
   capture the active =:id= as =:before-generation=.
3. Persist the planned transaction under
   =$XDG_STATE_HOME/nelix/transactions/=, or
   =~/.local/state/nelix/transactions/= when =XDG_STATE_HOME= is unset.
4. Execute command rows in order and rewrite the transaction record after every
   successful step with the current =:executed= rows.
5. On success, capture =:after-generation= and mark the record =:status ok=.
6. On command failure, write =:status error= with =:executed=,
   =:rollback-plan=, =:rollback=, and =:error=, then call =nelix-rollback= with
   =:before-generation= when it
   was captured.
7. After rollback, query the active generation again.  Rollback is reported as
   =:ok t= only when the active generation equals =:before-generation=.
8. Return failure to the CLI even when rollback succeeds, because the requested
   convergence did not complete.

Dry-run behavior:

- never queries generation history;
- returns =:transaction= metadata with =:dry-run t=;
- never writes a transaction record;
- shows whether rollback would be enabled for a real apply.

Safety invariants:

- *apply-before snapshot*: real Nix =apply= captures the active profile
  generation before the first install/remove command.  Real native =apply=
  captures the active native profile generation; if the profile does not exist
  yet, it creates an empty pre-apply generation so rollback has a concrete safe
  target.  If no command would run, no transaction record is written.
- *plan persistence*: the generated transaction record stores the exact
  =nelix-plan= plist that real apply is executing, including install/remove
  rows, command rows, lock metadata, remove safety metadata, and transaction
  preview state.
- *step journal*: after each successful command row, the record is rewritten
  with the current =:executed= rows.  A crash or command failure therefore
  leaves enough information to see the last confirmed mutation.
- *rollback plan*: every record stores a derived =:rollback-plan=.  It is
  available only when rollback is enabled and the before-generation snapshot
  was captured.
- *rollback disabled*: =apply --no-rollback= still writes a transaction record
  for real mutation attempts, but =:rollback-plan= has =:available nil= and
  =:reason "rollback-disabled"=.  On failure, =:rollback= records
  =:attempted nil= with the same reason and no rollback command is run.
- *rollback verification*: rollback success is not assumed from the rollback
  command exit alone.  =nelix apply= queries the active generation afterwards
  and reports =:verified t= only when it equals the captured
  =:before-generation=.  The installed CLI gate forces a Nix install failure,
  requires a profile rollback command, and checks the persisted transaction
  record for =:status error=, =:record-status error=, =:rollback=, and the
  successful pre-failure executed step.
- *dry-run equivalence*: =apply --dry-run= and real =apply= must be based on the
  same =nelix-plan= path.  Dry-run may omit generation capture and record
  writes, but it must expose the same install/remove/protected command intent
  that real apply would execute after remove-safety confirmation.  The
  installed CLI gate compares =plan --dry-run= and =apply --dry-run= JSON for
  =install=, =remove=, =keep=, =protected=, =commands=, =count=, and =empty=.
- *lockfile timing*: =apply --locked= validates and loads the lockfile before
  transaction begin and before any profile mutation.  Current apply does not
  rewrite the lockfile; lockfile creation, migration, and refresh are explicit
  =nelix lock= / =nelix lock migrate= operations.  Future lockfile update
  automation must happen only after a successful apply and must be journaled as
  a separate transaction step.

Opt-out:

#+begin_src sh
nelix apply MANIFEST --no-rollback
#+end_src

This is intended only for debugging.  Normal interactive and scripted use
should keep rollback enabled.

Current returned transaction shape:

#+begin_example
:transaction
(:enabled t
 :rollback-on-error t
 :generation-captured t
 :rollback-available t
 :before-generation 7
 :after-generation 8
 :record-id "apply-..."
 :record-file "~/.local/state/nelix/transactions/apply-....el"
 :record-status ok)
#+end_example

The transaction record is a generated S-expression plist with schema
=nelix-apply-transaction= version =1=.  It stores the planned command rows, the
latest executed rows, the transaction metadata, the rollback plan, and the
rollback result on failure.  This is the durable audit trail for "what was going
to happen" and "how far apply got" before rollback.

The durable records are inspectable through the CLI:

#+begin_example
nelix transaction list
nelix transaction list --limit 5
nelix transaction show apply-...
nelix --json transaction show ~/.local/state/nelix/transactions/apply-....el
#+end_example

=transaction list= returns summaries only, including record id, status,
manifest, command count, executed count, and rollback availability.
=transaction show= returns the validated record body under =:record= so that
the CLI status and the apply record status do not collide.
Record validation rejects schema v1 files missing the stable top-level,
=:plan=, =:transaction=, =:rollback-plan=, or executed-row keys needed for
post-failure diagnosis.

** 2.1 Transaction state machine

Transaction records have a deliberately small state machine:

#+begin_example
started -> running -> ok
started -> running -> error
started -> error
#+end_example

Stable status values:

- =started=: the plan snapshot and before-generation metadata were written
  before the first command row.
- =running=: at least one command row has been attempted and every row in
  =:executed= is known to have completed successfully.
- =ok=: all command rows completed, after-generation metadata was captured if
  available, and the apply result returned success.
- =error=: a command row failed, rollback was either attempted or explicitly
  unavailable, and the apply result returned failure.

There is intentionally no =rolled-back= top-level status.  Rollback is a
result attached to an =error= record because the requested convergence failed
even when recovery succeeded.  The durable answer to "did recovery work?" is
=:rollback (:attempted t :ok t :verified t ...)=.

The public schema command exposes these values:

#+begin_src sh
nelix --json schema transaction-v1
#+end_src

The schema output includes =status-values=, =rollback-plan-unavailable-reasons=,
and =rollback-result-keys=.  External tooling must treat unknown future values
as incompatible until the schema version changes.

** 2.2 Record write ordering

The record write order is part of the safety contract:

1. Validate manifest, imports, backend selection, remove confirmation, and
   locked-mode drift before transaction begin.
2. Capture =:before-generation= before the first mutating command.
3. Write =started= with the full plan and empty =:executed=.
4. After each successful command row, write =running= with the appended
   executed row.
5. On failure, compute/attempt rollback and write =error= with =:rollback= and
   =:error=.
6. On success, capture =:after-generation= and write =ok=.

If the process crashes between a profile mutation and the next record write,
the previous record remains the last confirmed safe point.  Recovery tooling
must therefore prefer the captured =:before-generation= and the plan's command
rows over guessing from current profile state alone.

** 2.3 Manual recovery contract

Automatic rollback is best-effort.  Operators must be able to recover from a
partially failed or interrupted apply without re-running the same mutation:

#+begin_src sh
nelix transaction list --limit 10
nelix --json transaction show APPLY-ID
nelix rollback GENERATION
#+end_src

Manual recovery uses =:rollback-plan :generation= from the record when
=:rollback-plan :available= is true.  When rollback is unavailable, the record
must explain one of the stable reasons:

- =transaction-disabled=
- =rollback-disabled=
- =before-generation-missing=

The =rollback-result-keys= schema entry lists the stable keys a failed apply may
write under =:rollback=.  =:verified t= is the only success proof; command exit
alone is not enough.

Failure signals =anvil-pkg-error= with:

- human message including =rollback=ok= or =rollback=not-ok=;
- =:executed= command rows that completed before failure;
- =:transaction= metadata;
- =:rollback= result with =:generation=, =:after-rollback-generation=,
  =:verified=, and =:reason= when rollback did not return to the captured
  generation.

** 2.4 Requirement coverage

The current transaction contract covers the operational safety requirements as
follows:

| Requirement | Contract |
|-------------+----------|
| apply前 snapshot | Capture =:before-generation= before the first mutating command.  Nix uses the active profile generation; =nelix-native= creates/captures a profile generation so rollback has a concrete target. |
| 実行 plan の保存 | Write the full =nelix-plan= plist into the transaction record at =started=, before any install/remove command. |
| 成功済み step の記録 | Rewrite the transaction record after each successful command row with appended =:executed= rows and =:status running=. |
| 失敗時 rollback plan | Derive =:rollback-plan= from =:before-generation= and =:rollback-on-error=.  On failure, attempt rollback when available and persist =:rollback=. |
| =--dry-run= と実 apply の差分保証 | =plan --dry-run= and =apply --dry-run= share the same plan path and gates compare =install=, =remove=, =keep=, =protected=, and =commands=.  Dry-run never writes transaction records or queries generation history. |
| lockfile 更新タイミング | =apply --locked= validates the lock before transaction begin.  =apply= does not rewrite locks; =nelix lock= and =nelix lock migrate= are the explicit lock mutation commands. |

Current implementation evidence:

- =test/nelix-manifest-test.el= covers successful transaction records,
  rollback-on-command-failure, disabled rollback, rollback verification
  mismatch, native rollback, and transaction record shape validation.
- =tools/nelix-lock-plan-apply-gate.sh= runs the public CLI with a fake Nix
  backend, verifies dry-run has no install/remove side effects, verifies real
  locked apply records an =ok= transaction, forces an install failure, requires
  =profile rollback --to-generation 7=, and checks the persisted =error=
  transaction record.
- =packaging/verify-installed-nelix-cli-gate.sh= repeats the public installed
  CLI checks from the packaged payload.

Open design edge:

- There is no automatic "resume interrupted transaction" command yet.  The
  durable recovery contract is manual: inspect =transaction show=, then run
  =nelix rollback GENERATION= using =:rollback-plan=.  A future
  =nelix transaction recover APPLY-ID= command must use the same record schema
  and must not infer recovery state from current profile contents alone.

* 3. Remove Safety

Real =apply= refuses unmanaged remove actions unless the caller explicitly
confirms the remove set.

Supported safety flags:

#+begin_src sh
nelix apply MANIFEST --allow-remove
nelix apply MANIFEST --allow-remove-count N
nelix sync MANIFEST --prune --allow-remove-count N
#+end_src

The exact count mode is preferred for scripts because it proves the caller saw
the current plan.  =apply --dry-run= never requires the flag; it reports
=:remove-safety= metadata so tools can show the required count before a real
apply.

Transaction rollback protects against command failure, not against applying an
undesired but valid remove plan.  Remove review remains mandatory.

Native transaction note: =nelix-native= installs can create one native profile
generation per package/dependency operation.  The transaction journal records
successful install steps and rolls the profile marker back to the captured
pre-apply generation if a later native install fails.  Store paths are not
deleted by rollback; they remain subject to native store GC, preserving the
same generation-based recovery model as the native store.

* 4. Debian Package Gate

The package now has an installed CLI gate:

#+begin_src sh
make verify-installed-cli-gate
#+end_src

It runs =/usr/bin/nelix= with a fake =nix= executable and verifies:

- =validate MANIFEST=;
- =lock MANIFEST= creates =MANIFEST.nelix-lock=;
- =lock validate MANIFEST= validates schema/version/format metadata without
  mutating profile or lock state;
- =lock diff MANIFEST= reports manifest/import drift against the lockfile
  without mutating profile or lock state;
- =lock migrate MANIFEST --dry-run= reports whether a compatible legacy lock
  needs a v2 rewrite, and =lock migrate MANIFEST= writes the current
  =MANIFEST.nelix-lock= while preserving the old file;
- =plan MANIFEST= reports install/remove rows;
- =apply MANIFEST --dry-run= reports transaction metadata without mutating.
- =apply MANIFEST --locked --allow-remove-count 1= enforces the lock and
  reaches the expected fake Nix profile install/remove commands without
  rewriting =MANIFEST.nelix-lock=.
- =transaction list= and =transaction show= can read the persisted apply record
  and report command counts, executed rows, and the rollback plan.

The source-tree NeLisp gate also runs =apply MANIFEST --locked --dry-run= so
locked mode is checked without loading the lock file as executable Elisp.

=make smoke-nelix-lock-schema= is the fast source-tree contract gate for
lockfile schema drift.  It runs =bin/nelix= with a fake =nix= backend and
checks that =docs/schema/nelix-lock-v2.schema.json=, =nelix schema lock-v2=,
=nelix schema all=, =nelix lock= JSON output, =lock validate=, =lock diff=,
and =lock migrate= agree.  It also verifies that read-only lock commands do
not rewrite =MANIFEST.nelix-lock=, that v1 legacy locks migrate to v2, and
that future schema versions are rejected.

=make deb-release-gate= now runs =deb-local-gate=, installs the freshly built
=.deb= with =install-built-deb=, then runs =verify-installed-debian=,
=verify-installed-cli-gate=, and =verify-user-environment=.  This ordering is
important because =deb-local-gate= rebuilds the package; installed CLI checks
must exercise that rebuilt payload, not a previously installed package with the
same version.  =debian/tests/load= also invokes the installed CLI gate so
autopkgtest covers the package-installed command path.

* 5. DSL v1 Contract

There are now two layers:

- low-level stable API: =(nelix-manifest :name ... :emacs ... :linux ...)=;
- user-facing v1 DSL: a thin, Nix/Guix-like declarative environment form.

The v1 DSL target syntax is:

#+begin_src emacs-lisp
(nelix-environment
 (name "madblack-21")
 (profile "default")
 (nix-channel "nixpkgs")
 (imports "custom-lisp/nelix-package-index.el"
          "custom-lisp/nelix-linux.el")
 (backend-policy
  (gnu/linux nix nelix-native apt dnf)
  (darwin nix nelix-native)
 (windows-nt nix))
 (emacs-packages nelix-package-emacs-packages)
 (linux-packages nelix-linux-base-nix-packages)
 (bootstrap-apt-packages nelix-linux-bootstrap-apt-packages)
 (package magit :backend elpa :group editor :feature git)
 (linux-package ripgrep :backend nix :pin t :platform gnu/linux)
 (version-pin fd "10.2.0")
 (remove-policy confirm))
#+end_src

Rules:

- =nelix-dsl.el= is the public require boundary for DSL v1; it must provide
  =nelix-environment= and report =nelix-dsl-version= as =1=;
- subform names are stable once v1 ships;
- each subform is single-assignment except repeated =package=,
  =linux-package=, and =version-pin= rows; invalid duplicates are hard errors
  in both the normal Emacs loader and the standalone NeLisp fast validator;
- plist =nelix-manifest= remains supported indefinitely for generated files and
  backward compatibility;
- package list subforms accept either a variable symbol or literal package
  symbols/strings;
- =package= and =linux-package= are first-class v1 environment rows.  They
  record metadata and also contribute their name to the normalized =:emacs= or
  =:linux= target list so existing plan/apply remains the only mutation path;
- package row options are =:backend=, =:pin=, =:version=, =:profile=,
  =:group=, =:feature=, =:platform=, and =:when=.  In v1, =:backend= is
  validated, =:pin t= contributes the package name to =:pins=, and the other
  options are stable metadata for lock/native-store evolution;
- =version-pin= rows are stored under =:version-pins= and contribute the
  package name to =:pins=.  Version resolution is not yet enforced by the Nix
  backend; lock/native-store resolution must consume the metadata before it
  becomes a hard version guarantee;
- =remove-policy= accepts =confirm=, =keep=, or =prune=.  It is recorded in the
  normalized manifest, but real mutation still requires CLI remove
  confirmation against the current plan;
- OS backend policy subforms are converted to the existing alist shape;
- supported backend symbols in DSL v1 are =nelix-native=, =nix=, =apt=,
  =dnf=, =git=, =elpa=, =homebrew=, =scoop=, and =winget=.  =nix= and
  =nelix-native= are profile-owning backends today; OS package managers are
  declared as bootstrap/fallback capabilities until their mutation backends are
  explicitly implemented;
- unknown subforms are hard errors;
- the DSL must expand to =nelix-manifest= so existing lock/plan/apply code stays
  the single implementation path;
- =nelix --json schema manifest-dsl-v1= must report schema
  =nelix-environment=, schema version =1=, the stable DSL form list, the
  form-to-manifest-key mapping, the stable backend name list, the accepted
  package row forms/options, the deferred v2 candidate forms, the forbidden
  private-data forms, and the v1 policies for removal review and private data.
- =nelix --json schema transaction-v1= must report schema
  =nelix-apply-transaction=, schema version =1=, and the required top-level,
  plan, transaction metadata, rollback-plan, rollback argv, and executed-step
  keys used by apply records.

Scope decisions for the user-requested Nix/Guix-style surface:

- package declarations in =nelix-environment= are desired-state rows, not full
  build recipes.  Native-store package recipes still live in the registry DSL
  =nelix-package=; registry recipes define version, source, install,
  dependencies, and per-system support.  Environment rows select and annotate
  those packages.
- Linux package declarations are represented by =(linux-package NAME ...)=
  rows and =(linux-packages ...)= lists for desired profile content,
  =(bootstrap-apt-packages ...)= for Debian bootstrap tools, and
  =(debian-tools ...)= for Debian packaging helper checks.
- backend selection is controlled by =(backend-policy ...)= and the registry
  recipe's per-system availability.  =:nix= / =:native= user wording maps to
  =nix= / =nelix-native= in the actual DSL.  DSL v1 accepts either a flat
  backend list such as =(backend-policy nix nelix-native)= or OS rows such as
  =(backend-policy (gnu/linux nix nelix-native dnf))=, and rejects unknown
  backend symbols at macro expansion time.
- pinning is =(pins ...)=, =:pin t= on package rows, and =(version-pin NAME
  VERSION)= for environment-level package-name holds.  Current profile
  mutation still pins by package name; version metadata is recorded for lock
  and native-store resolution.
- remove policy is stored in the manifest as =confirm=, =keep=, or =prune=,
  but real mutation still requires CLI confirmation via =--allow-remove= or
  =--allow-remove-count N= so review happens against the current plan.
- profile selection is =(profile "...")=.
- feature/group classification in v1 is available as package row metadata
  =:group= and =:feature=.  Imported Elisp variables remain the way to define
  reusable package groups.  First-class =(group ...)= and =(feature ...)=
  subforms are v2 candidates and are rejected as reserved in v1.
- platform conditions in v1 are split between =(backend-policy ...)= for
  environment-level OS policy, package row =:platform= metadata, and registry
  recipe =:systems= for package-level availability.  First-class =(platform
  ...)= forms are reserved for v2.
- secrets and private repositories must not be embedded in manifest or lockfile
  rows.  Use environment/configured credentials and registry roots/remotes with
  hash/signature policy.  Lockfiles may record public source identity and
  hashes, but not tokens.  DSL v1 rejects private-data forms such as
  =(secret ...)= and =(private-repo ...)= explicitly.

Implementation status: =nelix-environment= is implemented as a macro expanding
to =nelix-manifest=.  Existing plist manifests remain valid.  The Debian
installed CLI gate loads a real =nelix-dsl= manifest using imports, package
variables, OS-row backend policy, bootstrap package declarations, package row
declarations, version pins, remove policy, and pins, then runs =validate= and
=plan --dry-run= through the public command line.

* 6. Lockfile Schema Direction

Current lockfiles are Elisp/S-expression files with an explicit schema header:

#+begin_example
:schema "nelix-lock"
:schema-version 2
:version 2
:format sexp
#+end_example

CLI =--json lock= serializes the same data for external tooling.

The external JSON contract is fixed in:

#+begin_example
docs/schema/nelix-lock-v2.schema.json
#+end_example

This schema describes the JSON form emitted by =nelix --json lock=, while
=MANIFEST.nelix-lock= remains the source-of-truth S-expression file that Emacs
and NeLisp load directly.

Current compatibility rule:

- legacy v1 locks are readable for drift checks and can be migrated with
  =lock migrate=, but =apply --locked= refuses them before migration because
  they do not contain package rows;
- legacy v2 locks without explicit =:schema= / =:schema-version= are readable
  and schema-compatible;
- current v2 locks with explicit schema metadata are covered by a fixture with
  package rows;
- future schema versions are rejected by =lock-check= and =apply --locked=
  until support is added.

The regression suite commits fixture lockfiles under =test/fixtures= and
round-trips real =nelix --json lock= output through the standard JSON parser.
The installed Debian CLI gate also creates a legacy v1 lock, verifies that
=lock migrate --dry-run= is read-only, runs the real migration, and validates
the resulting v2 lock through the public CLI.

The CLI also exposes a compact machine-readable schema contract:

#+begin_src sh
nelix --json schema
nelix --json schema manifest-dsl-v1
nelix --json schema lock-v2
nelix --json schema transaction-v1
#+end_src

This command is read-only and generated from the runtime constants used by
=nelix-environment=, lockfile writing, and transaction record validation.
Package gates check it so downstream automation can rely on the DSL, lock, and
transaction version boundaries without parsing Elisp source files.

For =lock-v2=, the schema summary is itself the operational contract:

- =source-of-truth= is =MANIFEST.nelix-lock=.
- =json-output= is =nelix --json lock MANIFEST=.
- =commands= lists the public lock lifecycle commands: =lock=,
  =lock validate=, =lock diff=, and =lock migrate=.
- =compatibility= records the stable compatibility labels:
  =legacy-v1-readable-migrate-required=, =legacy-v2-readable=, and
  =future-version-rejected=.
- =migration=, =validation=, and =diff= name the exact CLI commands external
  tools should call instead of loading the S-expression file directly.

This keeps the S-expression lockfile as the runtime source of truth while
giving non-Elisp tooling a stable JSON command and a discoverable lifecycle
contract.

Do not replace the S-expression lockfile until NeLisp and Emacs both have a
stable JSON parser/writer path for this data.

* 7. Native Store and Wrapper Policy

=nelix-system-*= wrapper packages are transitional.  Replacement policy:

- keep wrappers in the standard Linux package set while Debian tools still come
  from bootstrap APT packages;
- replace each wrapper with a native Nix or Nelix-native recipe only when the
  command set is equivalent enough for package development gates;
- remove the wrapper from =nelix-linux-debian-wrapper-nix-packages= in the same
  change that adds the native recipe and updates docs/tests.

=ripgrep= is now part of the core Linux tool set because it is needed for
source inspection and package migration work.

* 8. Operational Docs

The user manual must keep a dedicated "real apply" section:

- run =validate=;
- run =lock=;
- run =plan=;
- run =apply --dry-run=;
- inspect =install/remove/commands/transaction=;
- run =apply= only after remove review;
- use =nelix rollback GENERATION= or the transaction error's rollback report
  when recovery is needed.
