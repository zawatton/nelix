* Status

INITIAL-DELIVERABLES-IMPLEMENTED.  This document defines the next
architectural step after Doc 20 and Doc 21: making Nelix usable on
Windows, macOS, and machines where Nix is not installed.

The initial deliverables are implemented:

- =nelix-backend.el=
- =nelix-store.el=
- =nelix-registry.el=
- =nelix-fetch.el=
- =nelix-builder.el=
- =test/nelix-store-test.el=
- local registry fixtures for =ripgrep= and =magit=
- packaged registry recipes for existing system commands =ripgrep=, =git=,
  =curl=, =fd=, =jq=, and =tree=
- =nelix-native-audit=, which works without Nix installed
- hash-verified local/file/HTTP/ELPA/Git fetch source path
- native =unpack= builder that extracts verified archives into the
  native store and creates profile generations

This implementation can install simple archive recipes into the native
store through =nelix-native-install-recipe= / =nelix-native-install= /
=nelix-backend-install=.  =nelix-apply= / =nelix-audit= now choose a
manifest backend and fall back to =nelix-native= when Nix is not
available.  Native installs preserve existing profile entries across
generations and resolve registry =:dependencies= before installing the
requested package, with dependency-cycle detection.

Native store MVP evidence is currently tracked as follows:

| MVP requirement | Evidence |
|-----------------+----------|
| recipe registry | =nelix-registry.el=, =test/nelix-store-test.el= registry tests, and =packaging/verify-nelix-native-cli-gate.sh= =registry list= / =registry index= checks |
| fetch | =nelix-fetch.el= and ERT coverage for =url=, =elpa=, =github-release=, =git=, and =local= sources |
| hash verify | =nelix-fetch-verify-file= / =nelix-fetch-verify-local-source= tests plus the CLI bad-hash install rejection gate |
| unpack | =nelix-builder.el= archive builders and the CLI =native install fixture-archive= gate |
| profile activation | =nelix-profile-activate-runtime= tests, explicit POSIX =symlink= mode coverage, and CLI checks for active shims, =path.sh=, and the =active/profile= file tree |
| rollback | =nelix-profile-rollback= tests plus the CLI =native rollback --generation= gate |
| lockfile recording | =nelix lock= records native =:recipe-source=, =:recipe-install=, =:recipe-dependencies=, and source hashes; the CLI gate replays =apply --locked= after registry drift |

The packaged gate prints
=nelix native store MVP ok: recipe-registry fetch hash-verify unpack profile-activation rollback lockfile-recording=
only after these CLI checks pass, and =verify-deb-contents= verifies that the
packaged verifier still contains that MVP anchor.

The intent is not to clone Nix byte-for-byte.  The intent is to
recreate the useful Nix mechanisms in Nelisp/Elisp:

- declarative package recipes
- source/substitute hashes
- content-addressed store entries
- immutable package outputs
- user profiles
- generations and rollback
- manifest + lock based reproducibility
- remote registry and optional binary substitutes

Existing Nix support remains valuable.  Nix should become one backend
among several:

#+begin_example
Nelix manifest
  ↓
resolver
  ↓
backend:
  - nix           existing Linux/macOS users with Nix installed
  - nelix-native Windows/macOS/Linux without Nix
  - external     apt/dnf/homebrew/winget/scoop as bootstrap or fallback
#+end_example

* Goal

Make this workflow possible on any supported OS:

#+begin_src shell
nelix apply ~/.config/nelix/manifest.el
nelix audit ~/.config/nelix/manifest.el
nelix upgrade-plan ~/.config/nelix/manifest.el
nelix sync ~/.config/nelix/manifest.el
#+end_src

With these constraints:

- The user should not need to install Nix.
- The same manifest should work across Linux, macOS, and Windows as
  far as package availability allows.
- Nelix should prefer immutable store entries over mutating random
  directories.
- Lock files must record exact sources and hashes.
- The initial package server can reuse public upstreams and selected
  Nix metadata, but Nelix must be able to migrate to its own registry
  and cache later.

* Non-goals

- Nix language compatibility.
- Full =nixpkgs= evaluation inside Nelisp.
- Full =cache.nixos.org= compatibility in the first version.
- Rebuilding all packages from source.
- Sandboxed build parity with Nix on Windows/macOS in the first
  native backend.
- System package manager replacement for kernel drivers, services,
  and privileged OS components.

* Backend Model

** Backend roles

| Backend | Purpose |
|---------+---------|
| =nix= | Use existing =nix profile= commands. Current implementation. |
| =nelix-native= | Fetch/build/unpack into Nelix store without requiring Nix. |
| =elpa= | Install or vendor Emacs Lisp packages from ELPA/MELPA-style archives. |
| =git= | Fetch source trees directly from Git. |
| =os= | Delegate bootstrap or fallback packages to apt/dnf/homebrew/winget/scoop. |

** Backend policy

Backend selection must be explicit and inspectable.

#+begin_src emacs-lisp
(setq nelix-backend-policy
      '((gnu/linux . (nelix-native nix apt git elpa))
        (darwin    . (nelix-native nix homebrew git elpa))
        (windows   . (nelix-native scoop winget git elpa))))
#+end_src

The policy is ordered.  The resolver tries the first backend that has
a recipe for the target and current system.

** Backend capability record

Each backend advertises capabilities.

#+begin_src emacs-lisp
(:backend nelix-native
 :systems (x86_64-linux aarch64-linux x86_64-darwin aarch64-darwin
           x86_64-windows)
 :fetchers (url git github-release elpa)
 :store t
 :generations t
 :rollback t
 :build nil
 :binary-substitutes t)
#+end_src

Audit must report backend availability before mutating anything.

* Store Layout

** Linux/macOS

#+begin_example
~/.local/share/nelix/store/
  sha256-abc123-ripgrep-14.1.1/
  sha256-def456-magit-4.3.0/

~/.local/state/nelix/profiles/default/
  current -> generations/42
  generations/
    41/
    42/
#+end_example

** Windows

#+begin_example
%LOCALAPPDATA%/nelix/store/
  sha256-abc123-ripgrep-14.1.1/
  sha256-def456-magit-4.3.0/

%LOCALAPPDATA%/nelix/profiles/default/
  current.json
  generations/
    41/
    42/
#+end_example

Windows must not rely on POSIX symlink behavior.  Profile activation
uses one of:

- shim scripts
- copied launchers
- directory junctions where available
- generated =PATH= fragments
- Emacs =load-path= augmentation for Elisp packages

** Store path naming

Initial native store path:

#+begin_example
<hash>-<name>-<version>
#+end_example

Example:

#+begin_example
sha256-lnv3k...-ripgrep-14.1.1
#+end_example

The hash is the normalized output hash when available.  If only a
source hash is known, the store entry must record that it is
source-addressed, not output-addressed.

* Store Entry Metadata

Each store entry has a metadata file.

#+begin_example
store/sha256-...-ripgrep-14.1.1/.nelix/store-entry.el
#+end_example

Shape:

#+begin_src emacs-lisp
(nelix-store-entry

 :name "ripgrep"
 :version "14.1.1"

   :repo "BurntSushi/ripgrep"
   :tag "14.1.1"
   :asset "ripgrep-14.1.1-x86_64-unknown-linux-musl.tar.gz"
   :sha256 "sha256-...")
 :output-hash "sha256-..."
 :files '("bin/rg" "share/man/man1/rg.1")
 :runtime-paths '("bin")
 :features nil
 :created-at "2026-06-17T13:00:00+09:00")
#+end_src

* Profile Model

A profile is an ordered set of store entries plus activation metadata.

#+begin_src emacs-lisp
(nelix-profile
 :schema 1
 :name "default"
 :generation 42
 :system x86_64-linux
 :entries
 '((:name "ripgrep"
    :store-path ".../sha256-...-ripgrep-14.1.1"
    :runtime-paths ("bin"))
   (:name "magit"
    :store-path ".../sha256-...-magit-4.3.0"
    :emacs-load-paths ("share/emacs/site-lisp/magit"))))
#+end_src

Activation writes:

- =PATH= fragments
- Emacs =load-path= fragments
- shell profile snippets where requested
- POSIX shell shims and Windows =.cmd= shim scripts or launcher directories
- an =active/profile= file tree: POSIX profiles use symlinks when
  available, and Windows profiles copy files so activation does not
  require privileged symlink creation

Activation is built in a temporary directory and committed by replacing
the previous =active= tree only after all shims, path fragments, and
profile links/copies have been written.  If activation fails before the
commit step, the previous =active= tree remains intact.

Rollback changes =current= or =current.json= to an earlier generation.
Store entries are garbage-collected only when no profile generation
references them.

** Emacs Lisp recipe

#+begin_src emacs-lisp
(nelix-package
 :name "magit"
 :version "4.3.0"
 :class emacs-lisp
 :source (:type elpa
          :archive gnu
          :package "magit"
          :version "4.3.0"
          :sha256 "sha256-...")
 :dependencies '("transient" "with-editor")
 :install (:type emacs-lisp
           :features (magit)))
#+end_src

** Recipe invariants

- Every network source must have a hash.
- Every binary substitute must have a hash and system id.
- Every package must declare class and system support.
- A recipe can be incomplete for some systems, but resolver must make
  that explicit.
- A recipe is data, not arbitrary code, unless explicitly marked
  =:trusted-code t=.

** Remote registry

Initial server can be static files.

#+begin_example
https://packages.nelix.example/index.json
https://packages.nelix.example/packages/ripgrep.el
https://packages.nelix.example/packages/emacs/magit.el
https://packages.nelix.example/substitutes/x86_64-linux/ripgrep-14.1.1.json

Required commands:

#+begin_src shell
nelix registry update
nelix registry index ~/.local/share/nelix/registry ./index.el
nelix registry list
nelix registry list --system x86_64-linux
nelix registry search ripgrep
nelix registry search ripgrep --system x86_64-linux
nelix registry show ripgrep
#+end_src

Status: implemented in the installed CLI as =nelix registry update=,
=nelix registry index ROOT OUTPUT=, =nelix registry list [--system SYSTEM]=,
=nelix registry search QUERY [--system SYSTEM]=, and =nelix registry show
NAME=.  List/search/show refresh the local registry cache before reading
recipes so the CLI behaves like a package-manager front end rather than a raw
in-memory Elisp helper.  The index command emits a static
=nelix-registry-index= file with recipe paths and SHA-256 hashes so a local
recipe tree can become a hash-verified remote registry.

Elisp API:

#+begin_src emacs-lisp
(nelix-registry-update)
(nelix-registry-write-index "~/.local/share/nelix/registry" "./index.el")
(nelix-registry-search "ripgrep")
(nelix-registry-get "ripgrep")
#+end_src

** Registry trust

Minimum trust model:

- registry root URL is configured by the user
- registry index has a hash in local state
- package recipes carry source hashes
- downloaded archives are verified before store insertion
- remote registry indexes can require a trusted signature through
  =:require-signature=, =:sig=, =:trusted-signers=, and =:public-keys=
- remote package recipe rows can require a trusted signature through the
  same row-level policy fields
- registry recipe files are parsed as data; only =require nelix-registry=
  and a single =nelix-package= form are accepted, and unsupported forms are
  rejected without evaluation

Future trust model:

- transparency log for recipe revisions
- multiple registry mirrors

Registry index and package recipe signatures reuse the native substitute
signature policy.  Index signed messages are produced by
=nelix-registry-index-signature-message= and cover the remote name, index
URL, index sha256, index schema version, and package count.  Recipe row
messages are produced by =nelix-registry-recipe-signature-message= and
cover the remote name, row path/url, recipe sha256, recipe name, version,
and class.  This keeps the initial static registry bootstrap OpenSSL-free
when using =nelix-sha256-digest=, while still allowing the same
RSA/Ed25519 verifier registry used by substitutes.

* Nix Metadata Reuse

The user goal includes using Nix package servers initially.  This is
possible only in carefully scoped ways.

** What can be reused early

- Package names and descriptions from =nix search --json= when Nix is
  available on a maintainer machine.
- Nixpkgs version metadata exported into Nelix registry recipes.
- Upstream URLs and hashes extracted from selected nixpkgs derivations.
- Precomputed substitute metadata for specific store paths, if a
  maintainer exports it into Nelix format.

** What should not be attempted first

- Evaluating arbitrary nixpkgs expressions inside Nelisp.
- Fetching =cache.nixos.org= artifacts by package name alone.
- Assuming Linux Nix store outputs work on Windows.
- Treating =narinfo= as a generic package metadata API.

** Nix substitute bridge

Nelix supports a precomputed Nix substitute metadata bridge:

#+begin_src emacs-lisp
(nelix-substitute
 :name "ripgrep"
 :version "14.1.1"
 :system x86_64-linux
 :source nix-cache
 :cache "https://cache.nixos.org"
 :store-path "/nix/store/..."
 :nar-hash "sha256-..."
 :nar-size 123456
 :references '(...)
 :sig "cache.nixos.org-1:...")
#+end_src

The bridge is intentionally scoped:

- it does not evaluate arbitrary nixpkgs expressions
- it does not query =cache.nixos.org= by package name
- it accepts maintainer-exported metadata only
- it requires explicit cache URL, Nix store path, nar hash, system, and
  package identity
- it uses the same substitute signature policy as native substitute
  metadata

N8 includes materialization and install:

- =nelix-substitute-materialize= downloads the payload, verifies the
  declared hash, restores true NAR payloads with =nix-store --restore=,
  and unpacks portable tar/zip payloads through the native builder.
- =nelix-substitute-install= creates a native profile generation from
  the materialized store entry.
- Registry/package servers can publish the same stable descriptor
  contract for native substitutes and maintainer-exported Nix cache
  metadata.

** Substitute signature policy

N8 implements signature policy and initial cryptographic verification:

#+begin_src emacs-lisp
(setq nelix-substitute-trusted-signers '("nelix.example-1"))
(setq nelix-substitute-public-keys
      '((:key "nelix.example-1"
         :algorithm openssl-rsa-sha256
         :public-key-file "/etc/nelix/keys/nelix.example-1.pem")))
(nelix-substitute-verify-trust substitute)
#+end_src

This confirms that a descriptor has a signature, that the key name is
trusted locally, and, when the descriptor declares
=:algorithm openssl-rsa-sha256=, that OpenSSL can verify the signature
over =nelix-substitute-canonical-message=.  Unsigned descriptors,
unknown signers, missing public keys, unsupported algorithms, and
invalid signatures are reported as blocked and must not be treated as
trusted substitutes.

Cryptographic verification is dispatched through
=nelix-substitute-crypto-verifiers=.  Native NeLisp crypto can register
future Ed25519/RSA verifiers without changing the trust-policy entry
points.  Nelix currently ships an OpenSSL-free
=nelix-sha256-digest= verifier that compares the SHA-256 digest of the
canonical descriptor message against a trusted signer entry.  This is a
native integrity verifier for bootstrap/cache experiments, not a
public-key signature replacement.

The Nix bridge also parses =.narinfo= text and converts it into the
same substitute descriptor contract.  =nelix-substitute-narinfo-fingerprint=
builds the Nix valid-path signature message:

#+begin_example
1;<store-path>;<nar-hash>;<nar-size>;<comma-separated full references>
#+end_example

Narinfo =Sig:= rows are recognized as Nix Ed25519 signatures.  The
current verifier accepts Nix public keys in =key-name:base64(raw32)= or
raw base64 form, wraps them as Ed25519 SubjectPublicKeyInfo DER, and
verifies the signature with OpenSSL =pkeyutl -verify -rawin=.  Invalid
or tampered fingerprints are blocked as
=:cryptographic-signature-invalid=.

Remaining future bridge work:

- native nelisp public-key crypto backend without OpenSSL dependency
- reference recording
- path relocation policy

Because Nix outputs may contain embedded =/nix/store= paths, this
bridge is a later optimization, not the foundation of native Nelix.

* Fetchers

Initial fetchers:

| Fetcher | Use |
|---------+-----|
| =url= | Direct tarball/zip downloads. |
| =github-release= | Cross-platform binary release assets, with optional =:base-url= for GitHub-compatible mirrors. |
| =git= | Source checkout at exact rev. |

Required primitives in Nelisp:

- HTTP GET binary
- streaming download to file
- SHA-256 hashing
- archive extraction: tar.gz, tar.xz, zip
- zip archives with top-level directories can use =:strip-components= in the
  native builder, matching the tar recipe shape used for GitHub release
  assets.
- file permissions on Linux/macOS
- executable bit handling or shim generation on Windows

* Resolver

The resolver turns manifest targets into concrete install plans.

Input:

#+begin_src emacs-lisp
(nelix-manifest
 :name "default"
 :emacs '(magit)
 :linux '("ripgrep")
 :windows '("ripgrep")
 :backend-policy '((windows . (nelix-native scoop winget))
                   (gnu/linux . (nelix-native nix))))
#+end_src

Output:

#+begin_src emacs-lisp
(:system x86_64-linux
 :targets
 ((:name "ripgrep"
   :backend nelix-native
   :recipe ".../packages/ripgrep.el"
   :version "14.1.1"
   :source-hash "sha256-..."
   :dependencies nil)
  (:name "magit"
   :backend nelix-native
   :recipe ".../packages/emacs/magit.el"
   :version "4.3.0"
   :dependencies ("transient" "with-editor"))))
#+end_src

Resolver rules:

- Respect backend policy order.
- Prefer locked versions when lock exists.
- Error on ambiguous provider unless policy resolves it.
- Report unsupported systems as audit errors, not silent skips.  Implemented
  for manifest-requested native targets: =nelix-audit= passes the desired
  target list into =nelix-native-audit=, which returns
  =:unsupported-systems= rows and marks the audit not OK when a requested
  recipe lacks current-system data.
- Always produce an install plan before mutation.

* Lock File

Doc 21 lock files now record a combined manifest/import digest,
per-file digest rows, backend/system metadata, and resolved targets.
Native Nelix still needs stronger source/build entries before network
source replay can be treated as fully reproducible.

#+begin_src emacs-lisp
(nelix-lock
 :version 2
 :manifest-digest "sha256-..."
 :manifest-files
 '((:role manifest
    :path "/home/me/.config/nelix/manifest.el"
    :digest "sha256-...")
   (:role import
    :path "/home/me/.emacs.d/custom-lisp/nelix-package.el"
    :digest "sha256-..."))
 :backend nelix-native
 :system x86_64-linux
 :registry
 '((:url "https://packages.nelix.example/index.json"
    :revision "2026-06-17T00:00:00Z"
    :sha256 "sha256-..."))
 :packages
 '((:name "ripgrep"
    :version "14.1.1"
    :backend nelix-native
    :source (:type github-release
             :url "https://github.com/..."
             :sha256 "sha256-...")
    :output-hash "sha256-..."
    :store-path "sha256-...-ripgrep-14.1.1")
   (:name "magit"
    :version "4.3.0"
    :backend nelix-native
    :source (:type elpa
             :archive gnu
             :sha256 "sha256-...")
    :dependencies ("transient" "with-editor"))))
#+end_src

Lock policy:

- Apply may run without lock in early versions.
- =apply --locked= and =sync --locked= require a clean Doc 21 lock
  check and package row enforcement before mutation.  Native rows
  replay source/install/dependencies from the lock file without
  trusting the current registry.
- Native install from network must require source hash.
- Audit warns on manifest/import digest drift.
- Sync may refuse to prune when lock and manifest disagree unless
  =:allow-lock-drift t= is supplied.

* Security Model

Minimum:

- verify hashes before store insertion
- never execute downloaded code during recipe parsing
- recipes are data by default
- warn when a recipe requires arbitrary build code
- materialize/rebuild directly from native lock source/install rows
  without trusting the mutable registry after lock generation
- replay source-free =script-shim= lock rows because the install metadata is
  the complete recipe for generated wrapper packages
- separate temporary download/build directory from final store path
- verify remote registry index signatures when policy requires them
- verify remote package recipe signatures when policy requires them

Future:

- sandboxed builders
- per-backend trust policy
- binary transparency log
- reproducible build attestation

* Windows Notes

Windows needs first-class design, not Linux assumptions.

Key decisions:

- Store paths live under =%LOCALAPPDATA%=.
- Profile activation uses generated shims and JSON metadata.
- Executables may be =.exe=, =.cmd=, or PowerShell wrappers.
- Symlink creation may require privileges; avoid depending on it.
- PATH activation should be shell-specific:
  - PowerShell
  - cmd.exe
  - Git Bash/MSYS2
  - Emacs process environment
- Line endings must not affect hashes unless normalized by builder.

Initial Windows package classes:

- GitHub release binaries
- zip archives
- Emacs Lisp packages
- Scoop/winget fallback declarations

* macOS Notes

macOS can share more with Linux:

- symlinks are available
- executable bits matter
- aarch64/x86_64 split is important
- quarantine attributes may need cleanup for downloaded binaries

Initial macOS package classes:

- GitHub release binaries
- tar.gz/zip archives
- Emacs Lisp packages
- Homebrew fallback declarations
- Nix backend when Nix is installed

** Phase N4 — unpack builder

Status: implemented as initial =unpack=, =copy=, and =script-shim=
builders in =nelix-builder.el=.  The =copy= builder verifies a local
=:source (:type local :path ... :sha256 ...)= file/directory first,
then copies either the whole source or explicit =:files= mappings into
the native store.  The =script-shim= builder creates fetch-free
store entries from =:command= and =:target= metadata, emitting POSIX
shell shims or Windows =.cmd= wrappers into =bin/=.  Runtime package
entries preserve =:runtime-bins=, and =nelix-profile-activate-runtime=
regenerates profile-local
=active/bin= shims plus =path.sh= or =path.cmd= activation fragments.
Native install creates additive profile generations: an install replaces
the same package name when present, otherwise it keeps prior entries and
appends the new one.  Registry =:dependencies= are installed first so the
final generation contains both dependency entries and the requested package.
Native lock files include dependency closure rows, recipe source metadata,
recipe install metadata, and source hashes.  Locked apply replays those rows
without consulting the current mutable registry, so drifted registry recipes
cannot silently change a locked native apply.
It also materializes =active/profile= as a Nix/Guix-like profile tree:
POSIX systems use symlinks in =auto= mode when available, while
Windows profile systems use copied files.
Native builders populate a =.tmp-*= store build directory first, write
store-entry metadata there, and only then commit it to the final
store path.  Failed fetch/verify/unpack/copy/script-shim installs do
not expose partial store entries and do not create profile generations.
Activation is staged under =active.tmp-*= and only then swapped into
=active=, so a failed activation does not remove the previously active
runtime profile.

- Install simple binary archives into store.
- Copy verified local file/directory sources into store.
- Generate runtime path/profile metadata.
- Support Linux/macOS executable bit handling.
- Support POSIX shell shims and Windows =.cmd= shims.
- Install fetch-free =script-shim= packages into the native store.
- Preserve existing profile entries and install registry =:dependencies=
  before requested native packages.
- Reject native dependency cycles before mutating the repeated package.
- Commit native store entries from temporary build directories only after
  payload and metadata generation have succeeded.
- Support profile link/copy tree activation without requiring symlinks
  on Windows.
