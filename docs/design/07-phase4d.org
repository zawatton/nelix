* Locked decisions (continuing L18-L22 from doc 06)

** L23. MELPA recipe synthesis via =postUnpack=

When =build-system= is =(emacs-package :format "melpa")= and the
=:source= directory does NOT contain =recipes/<pname>= (or no
=recipes/= directory at all), the renderer adds a =postUnpack= phase
to the flake that synthesises a minimal recipe before =melpaBuild=
runs.

Synthesised recipe shape:

#+begin_src lisp
;; recipes/<pname>
(<pname> :fetcher git :url "<source-url>" :files ("*.el"))
#+end_src

Behaviour:

- =:source= type =github-fetch= → =:url= becomes the github URL.
- =:source= type =git-fetch= → =:url= becomes the upstream URL.
- =:source= type =url-fetch= → =postUnpack= is skipped (tarball
  layout cannot reliably be re-pinned via :fetcher).  The user must
  supply an explicit recipe file via =:melpa-recipe= keyword (new
  in Phase 4-D for this case).
- =:files= defaults to =("*.el")= unless the IR carries a
  =:melpa-files= override (also new keyword).

Detection heuristic:

- At render time, peek at the source URL via L24's pre-fetch path
  if available; otherwise assume "no recipe" and synthesise.  False
  positives (synth runs over an existing recipe) are harmless because
  the synth file lands in =recipes/= and =melpaBuild= picks the
  first match — but we still attempt to skip if a recipe is found
  via the pre-fetch lookup.
- Add a =:melpa-synth= keyword on =emacs-package= (default =auto=,
  values =auto= / =force= / =never=) so users can override.

Implementation lives in =anvil-pkg-dsl.el= as a renderer extension
(=anvil-pkg--synth-melpa-recipe=) plus an upstream check helper in
=anvil-pkg-emacs.el= (=anvil-pkg-emacs-has-recipe-p=) that reuses
L18 / L24 fetch infrastructure.

** L24. L18 pre-fetch coverage for =url-fetch= and =git-fetch=

L18 currently dispatches only for =github-fetch=.  Phase 4-D adds
two new dispatch arms.

*** L24a. =url-fetch= tarball header scrape

For =:source (url-fetch :url "..." :sha256 "...")=:

1. Check L26 deps cache (key = =sha256:<hash>=) — return on hit.
2. Download the tarball into a tmp file via =anvil-pkg-compat-http-get=
   (extended in 4-D to stream binary bodies — see contract).
3. Use =tar -tzf= to enumerate entries, find =<pname>-pkg.el= or
   =<pname>.el= in the top-level dir, =tar -xzOf= to extract that
   one file's contents.
4. Apply the existing L18 parser (=anvil-pkg-emacs--parse-define-package=
   / =--parse-package-requires-header=).
5. Cache result + cleanup tmp file.

Failure modes degrade to empty deps + warning, same as L18.  Tarball
download timeout: 30s (vs L18's 5s; tarballs are bigger).  Maximum
tarball size: 50 MiB (refuse + warn for larger; users with huge
tarballs should declare deps explicitly).

*** L24b. =git-fetch= shallow-clone scrape

For =:source (git-fetch :url "..." :rev "..." :sha256 "...")=:

1. Check L26 deps cache (key = =git:<url>@<rev>=).
2. =git clone --depth 1 --branch <rev> <url> <tmpdir>= (fall back
   to fetch + checkout if =--branch= rejects a SHA).
3. Read =<pname>-pkg.el= or =<pname>.el= from =<tmpdir>=.
4. Apply L18 parser.
5. Cache + remove =<tmpdir>=.

Cleanup uses =unwind-protect= so failed clones still remove the
tmpdir.  Clone timeout: 60s.  =--depth 1= keeps the disk hit small;
worst case for a fat repo is the single-commit working tree.

The dispatch lives in =anvil-pkg-emacs.el='s
=anvil-pkg-emacs-derive-deps= (which already takes the source plist
as input — Phase 4-D extends the =:type= dispatch from one arm to
three).

** L25. Per-package rollback (=pkg-rollback-package=)

New public function in =anvil-pkg.el=:

#+begin_src elisp
(pkg-rollback-package pkg-name)
;; → t on success, signals `anvil-pkg-error' if pkg not in current
;;   generation.
#+end_src

Behaviour:

1. Read current generation's package list from L26's mirror.
2. Synthesise a NEW generation = current minus =pkg-name=.
3. Re-render the flake.nix for the pruned set.
4. =nix profile add <flake>= (= L20's dispatch path) — Nix records
   this as a new generation, NOT as a rollback.
5. Refresh the L26 mirror so subsequent =pkg-history= sees the
   removal event.

Implementation note: the synthesised flake reuses each remaining
package's IR from the registry (=anvil-pkg--registry= hash-table —
already populated by =pkg-define=).  A package installed by name
without =pkg-define= (= straight nixpkgs lookup, no IR) cannot be
rebuilt this way; Phase 4-D refuses with =anvil-pkg-error= +
suggestion to use whole-profile =pkg-rollback=.

The =pkg-rollback= (whole-profile) path is unchanged.

** L26. =anvil-pkg-state= integration (cross-session caches)

Phase 4-C stored three caches in session-only =defvar= hash-tables:

| Variable                          | Purpose                            |
|-----------------------------------+------------------------------------|
| =anvil-pkg-emacs--deps-cache=     | L18 derived =:depends-on=          |
| =anvil-pkg--nix-version-cache=    | =nix --version= result             |
| =anvil-pkg--generations-cache=    | =nix profile history --json= mirror|

Phase 4-D promotes all three to =anvil-pkg-state= KV namespaces:

| Namespace                         | Key shape                                    |
|-----------------------------------+----------------------------------------------|
| =anvil-pkg:emacs-deps=            | =github:<owner>/<repo>@<rev>= /              |
|                                   | =sha256:<tarball-hash>= / =git:<url>@<rev>=  |
| =anvil-pkg:nix-version=           | constant key =default= (single value)        |
| =anvil-pkg:generations=           | constant key =mirror= (whole list)           |

Migration semantics:

- =anvil-pkg-state= becomes a /hard/ dependency of =anvil-pkg= — no
  more "promote when integration lands" comments.  =Package-Requires=
  (or =require=) declares it.
- On first call after upgrade, the session =defvar= caches stay
  populated for the current session but are no longer authoritative
  — every cache read goes to =anvil-pkg-state= first, falls back to
  the defvar (treated as a transient L1).
- =pkg-clear-cache= drops both: the =anvil-pkg-state= namespaces AND
  the session defvars.
- TTL for L18 deps: 30 days (matches Phase 4-C's
  =anvil-pkg-emacs--deps-ttl=).  TTL for nix-version: 1 day.
  Generations mirror: no TTL — refreshed on every install / list /
  rollback (= unchanged from 4-C).

Implementation: replace the three =defvar= read/write sites with
=anvil-pkg-state-get= / =anvil-pkg-state-put= calls.  Tests that
relied on =defvar= reset between cases switch to mocking
=anvil-pkg--call-state-fn= (new injection point — same pattern as
=anvil-pkg--call-nix-fn=) so we never touch the real SQLite during
ERT.

* Open questions

** OQ15. =anvil-pkg-state= cross-runtime portability

L26 makes =anvil-pkg-state= a hard dep, but the actual state library
lives in =anvil.el= and assumes Emacs's built-in =sqlite-*=
functions.  NeLisp does not yet expose =sqlite-*=.  Until Phase 5
ships a NeLisp SQLite backend, =anvil-pkg-state= on NeLisp falls
back to a plain alist persisted to JSON in =~/.local/state/=.  Same
API, lower performance — acceptable for the tiny anvil-pkg cache
shapes (< 1k entries).

** OQ16. Tarball size threshold

L24a's 50 MiB cap is a guess.  Real-world Elisp packages average
< 1 MiB.  Keeping the cap forces a graceful fail mode for someone
who points =url-fetch= at e.g. a CI artifact that bundles
non-Elisp.  If users complain, the cap moves to a defcustom
(=anvil-pkg-tarball-max-bytes=).

** OQ17. =melpa-synth= file glob heuristics

The synthesised recipe's =:files= defaults to =("*.el")=.  Many
real MELPA recipes use richer globs (=("*.el" "lisp/*.el"
"snippets")=).  Phase 4-D punts on auto-detection — users who need
non-default globs supply =:melpa-files= explicitly.  A future phase
could attempt detection by inspecting the source layout, but the
heuristic surface is large enough to defer.

** OQ18. Per-package rollback ordering

L25 synthesises a new generation by removing one package, but the
remaining packages are re-added in registry-iteration order.  If
Nix is sensitive to dependency ordering (it generally isn't —
=buildEnv= is order-agnostic), this is fine.  Smoke test in 4-D
will validate by inspecting the resulting profile's contents.

** OQ19. State migration from session caches

Users running Phase 4-C in a long-lived Emacs and upgrading to 4-D
will have populated session =defvar= caches.  Phase 4-D's first
call after upgrade copies them into =anvil-pkg-state= via a one-shot
migration helper (=anvil-pkg--migrate-session-caches=).  Migration
is silent (no prompt) because the session cache content is already
valid — it just moves to persistent storage.

** Landing order

D first (= sequential, lands on main alone), then A / B / C in
parallel worktrees off the post-D HEAD.

Rationale: A, B, C all want to write to / read from the persistent
state layer that D introduces.  If we land A / B / C first they
each implement their own ad-hoc persistence, then D refactors them
all — wasted churn.  D is also the smallest behavioural change (=
moves data from defvar to KV with same semantics) so it merges
cleanly without conflicting with the others.

Worktree base HEAD reminder (= Phase 4-B / 4-C lesson): when
spawning parallel agents, branch from the latest HEAD AFTER D
lands, not from the design commit.  Capture the post-D commit SHA
in each agent's prompt so the agent's ERT baseline is reported
correctly.
