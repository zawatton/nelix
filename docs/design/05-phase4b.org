* STATUS

Phase 4-B — DESIGN.  Builds on Phase 4-A (commits =089a966= →
=7c466b6= + regression tests =54cb18d=) which shipped the
=emacs-package= build-system on top of =pkgs.emacsPackages.trivialBuild=

Phase 4-B closes the three open questions Phase 4-A deferred (OQ4

Implementation will again be split into parallel codex sub-tasks

* Goal

Round out the =emacs-package= surface so it is a complete
replacement for =async-installer= for Nix-having users:

#+begin_src elisp
;; Phase 4-A could already do:
(pkg-define magit
  (version "3.3.0")
  (source (github-fetch :owner "magit" :repo "magit"
                        :rev "v3.3.0" :sha256 "sha256-..."))
  (build-system emacs-package)
  (depends-on (list dash transient with-editor)))
(pkg-install 'magit :require 'magit)

;; Phase 4-B adds:
(pkg-define magit
  (version "3.3.0")
  (source (github-fetch :owner "magit" :repo "magit"
                        :rev "v3.3.0" :sha256 "sha256-..."))
  ;; melpaBuild + Package-Requires header scrape:
  (build-system (emacs-package :format "melpa"
                               :native-comp t)))
;; depends-on derived from the package's own Package-Requires header.

;; And async installs with a sentinel callback:
(pkg-install 'magit
             :require 'magit
             :async t
             :on-success (lambda (_) (message "magit ready")))
#+end_src

Plus a migration helper for users with an existing
=async-installer-git-list= variable:

#+begin_src elisp
(anvil-pkg-import-async-installer
 :var 'async-installer-git-list
 :emit (expand-file-name "my-pkgs.el" user-emacs-directory))
;; → writes a file of pkg-define forms covering every entry.
;; Caller adds (load "my-pkgs.el") to init.el.
#+end_src

* Non-goals

- Replacing Phase 4-A's =trivialBuild= path.  =:format "trivial"=
  remains the default and the sole codepath when =:format= is
  omitted.  =melpa= is opt-in.
- Header scraping for non-Emacs packages.  =Package-Requires= is
  Elisp-specific; only =build-system emacs-package= entries
  participate.
- Bidirectional sync with =async-installer-git-list=.  The importer
  is a one-shot snapshot; future edits go to the generated
  =pkg-define= forms.
- Cross-package version solving.  We honour the user's pinned
  versions verbatim — no SAT solver, no overlay.  nixpkgs already
  pins the rest.

* Locked decisions (continuing L7-L12 from doc 04)

** L13. =:format= switch on =emacs-package=

Args form takes one new keyword:

#+begin_src elisp
(build-system (emacs-package :format STRING))
#+end_src

Where =STRING= is one of =\"trivial\"= (default, current behaviour) or
=\"melpa\"= (renders =pkgs.emacsPackages.melpaBuild= instead).
Anything else raises =anvil-pkg-dsl-error= at parse time.

Rationale: the two builders diverge on (a) byte-compile invocation
and (b) =Package-Requires= header parsing.  =melpa= is the right
choice when the upstream is a real MELPA recipe; =trivial= for a
loose collection of .el files.

** L14. =:native-comp= toggle on =emacs-package=

#+begin_src elisp
(build-system (emacs-package :format "melpa" :native-comp t))
#+end_src

Renders as a wrapped Emacs derivation
(=emacsPackagesNg.emacsWithPackages= style — exact attr path
selected at render time so the doc tracks nixpkgs).  Default is
=nil= because (a) Phase 4-A users already shipped without it, (b)
native-comp pins the Emacs version into the closure and inflates
the store, and (c) opt-in matches user preference for slim
profiles.  Phase 4-C may flip the default once we have measured
real-world build times.

** L15. =:depends-on= auto-derivation from =Package-Requires=

When the user does NOT supply =(depends-on ...)= and the
build-system is =(emacs-package ...)=, render time tries to read
=Package-Requires= from the source tarball / cloned repo and emit
the corresponding =packageRequires= attr automatically.

Behaviour:

- Explicit =(depends-on (list ...))= always wins.  No merging.
- Auto-derive only fires for =emacs-package=, never for stdenv /
  rust / python / go.
- Lookup tries (in order): =<pname>-pkg.el= → =<pname>.el= header
  → =melpaRecipes/<pname>= (last-resort, only when source =:type=
  is a github recipe).
- On miss: render with empty =packageRequires=, log a warning,
  proceed.  The Nix builder will still succeed for self-contained
  packages.
- Non-Elisp build-systems with =depends-on= raise at parse time
  (existing Phase 4-A invariant — unchanged).

The derivation runs at flake.nix render time (= during
=pkg-install=, before nix is invoked), not at =pkg-define= time,
because the source archive may not yet exist when =pkg-define=
runs.  Implementation lives in =anvil-pkg-emacs.el= (new file) so
=anvil-pkg-dsl.el= stays pure.

** L16. =:async= keyword on =pkg-install=

Synchronous remains the default for =pkg-install=.  =(:async t)=
opts in to a non-blocking install:

#+begin_src elisp
(pkg-install 'magit
             :async t
             :on-success (lambda (result) ...)
             :on-error   (lambda (err) ...))
;; Returns a process object; nil callbacks are tolerated.
#+end_src

Sentinel pattern follows =async-installer='s proven approach: a
single =make-process= per install with a =:sentinel= that, on
=finished\n=, runs the post-install hook (load-path augment +
=:require=) and then the user's =:on-success=.  On non-zero exit
the sentinel collects =:stderr=, signals
=anvil-pkg-nix-failed= inside the user's =:on-error= (or via
=run-with-timer 0= when no =:on-error= was supplied so the error
still surfaces in =*Messages*=).

NeLisp standalone has its own process model — Phase 4-B leaves
=:async t= unsupported on NeLisp (signals
=anvil-pkg-async-not-supported=) and revisits in Phase 5 once
=anvil-pkg-compat= grows an async primitive.

** L17. =anvil-pkg-import-async-installer= = read-only utility

The importer scans =async-installer-git-list= (or any caller-
supplied list with the same shape) and emits =pkg-define= forms.
It does NOT:

- mutate the user's =async-installer= state;
- call =pkg-install= on the imported entries;
- back up or remove the existing =async-installer= setup.

Output is a single Elisp file; user controls when (and whether) to
load it.  Idempotent: re-running the importer overwrites the
output file with current state.

Lives in =anvil-pkg-import.el= (new file) so the converter is
loadable independently of =anvil-pkg-emacs.el='s Package-Requires
scraper — users who only want migration assistance don't pay for
the scraper require.

* Open questions

** OQ7. =melpaBuild='s recipe-file requirement

=melpaBuild= expects a MELPA recipe file (=recipes/<pname>=) in
the source tree.  Many GitHub repos are not MELPA-shaped.  Phase
4-B's first cut: when =:format "melpa"= is requested but no recipe
is found, synthesize a minimal one (=(<pname> :fetcher git :url
"...")= = inferred from =:source=) and inject it via a Nix
=postUnpack= hook.  Risky?  Maybe.  Phase 4-C may add a
=:recipe= keyword for explicit override.

** OQ8. Header scraper byte budget

=Package-Requires:= can sit anywhere in the first ~20 lines of a
.el file.  Reading the whole file is wasteful for large libraries.
Phase 4-B reads the first 4 KiB only; if the marker is not in
those bytes, treat as miss.  4 KiB matches what =package.el=
itself uses.

** OQ9. Async install on Windows (msys / native)

=make-process= on Windows native Emacs has known stderr buffering
quirks.  Phase 4-B implements assuming Linux/macOS semantics; a
Windows fix lands in Phase 4-C if real-world usage flushes the
issue out.  Smoke test runs on Linux only for now.

** OQ10. Caching the scrape result

Successive =pkg-install= of the same package re-scrapes the
header.  We could cache by =:source= sha256, but the cost is
negligible (≪ 1 ms) and the cache invalidation logic is a
bug-magnet.  Phase 4-B does NOT cache; revisit if profile builds
become I/O bound.
